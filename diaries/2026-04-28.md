## Mental model shift — Unity LEGOs vs UE5 recipes

> "all this time ive bene using plaugins fromt eh unity assets store and all drag and drop all mickey mouse land but now when UE 5 came we are modifying exisiting plugin code"

Captured the model:

**Unity:** plugins are precompiled `.unitypackage` LEGO pieces. Drag in, call from C#, never touch the source.

**UE5:** plugins are C++ source folders that get compiled INTO your project's binary at build time. The plugin is part of YOUR build, not a separate library. You can read its source, modify it, fix bugs.

```
HoloBornUE5/
├── HoloBornUE5.uproject
├── Source/HoloBornUE5/         ← your game code
└── Plugins/
    ├── glTFRuntime/            ← entire repo dropped in
    └── AndroidCamera2Plugin/   ← tark146 fork dropped in
```

When UE5 builds: walks `Plugins/`, compiles every `.cpp`, links it all into one big binary that ships in the APK. From Blueprint or C++ in your game code, you call functions like `glTFLoadAssetFromUrl` because they're decorated with `UFUNCTION(BlueprintCallable)` — that macro auto-exposes C++ to Blueprint.

For HoloBorn: 90% Blueprint (state machine, GLB load, spawn, polling, HUD). One small C++ class for HTTP multipart upload (Blueprint doesn't have a multipart builder).
