It was not.

### Round 1 — pink avatar (the URP shader strip surprise)

First press: avatar spawned, status text turned green (`Spawned at 1.9, 0.0, 0.1`), but the avatar rendered as a small pink/magenta humanoid silhouette. Two issues stacked:

1. **Pink** = `Hidden/InternalErrorShader` — Unity's URP build pipeline had stripped glTFast's PBR shader variants because no scene material referenced them statically. At runtime glTFast tried to bind textures to a shader that wasn't in the APK → error fallback → magenta.
2. **Tiny** = TRELLIS GLBs come out normalized to a unit cube (~1m tall raw), needed scale-to-human-height + floor-align logic that the legacy `LoadAndInstantiateGLB` had but my fresh `TestGlbLoader` lacked.

Added scale + floor-align (`Vector3.one * 1.7f` / `bounds.min.y` offset). Easy fix. Then started the shader rabbit hole.

### Round 2 — Always Included Shaders picker can't find package shaders

First instinct (correct, per Unity docs in retrospect): add `Shader Graphs/glTF-pbrMetallicRoughness` to Project Settings → Graphics → Always Included Shaders. Open the picker, type `glTF-pbrMetallicRoughness`. **Picker returns "None."**

The picker only searches `Assets/`, not `Packages/`. The glTFast shadergraphs live in `Library/PackageCache/com.unity.cloud.gltfast@.../Runtime/Shader/`. Same friction-hit yesterday with package shaders for the IDE-vs-Inspector wiring, just hadn't internalized it as a class of bug.

Pivoted to a runtime fallback: walk all renderers post-spawn, swap any material to URP/Lit, preserve textures via property-name probing. Wrote `ForceURPLitMaterials()`.

### Round 3 — URP/Lit also stripped, runtime fallback bails on line 1

Built, sideloaded, pressed B. **Still pink**. Logs revealed why:

```
E/Unity: [TestGlbLoader] URP/Lit not found — fallback impossible (URP active?)
```

`Shader.Find("Universal Render Pipeline/Lit")` returned NULL. URP/Lit got stripped too — the project genuinely had zero materials referencing it at compile time (HUD canvas uses UI shaders, all visible content is Sprites/Default from the legacy AR-guides era). Build pipeline was being aggressive.

Updated the runtime swap to try a list of fallback shaders: URP/Lit → URP/Unlit → Sprites/Default → Standard → Unlit/Texture. Whichever one survived the strip wins. **Sprites/Default was guaranteed in the build** (legacy doc confirmed it was added to Always Included Shaders during the AR-guides era, and the user's Project Settings screenshot showed it as Element 4 of the array).

### Round 4 — `forced 0 materials` (the real smoking gun)

Built, sideloaded, pressed B. **Still pink**. Logs:

```
[TestGlbLoader] using fallback shader: 'Sprites/Default'
[TestGlbLoader] forced 0 materials to 'Sprites/Default'
[TestGlbLoader] original world bounds: size=(0.40, 0.99, 0.42), renderers=1
```

**Zero materials swapped.** Renderer existed (bounds non-zero, mesh present), but `r.sharedMaterials.Length == 0` — glTFast hadn't actually assigned any materials to the renderer, so my "iterate existing materials and swap them" loop never executed.

Investigated whether this was a glTFast async lifecycle thing. Updated the swap path to:
1. Add `await Task.Yield()` x2 after `InstantiateMainSceneAsync` to give Unity a frame
2. Force-assign a fresh `Material[]` array sized by submesh count, regardless of whether old materials existed

This was the moment user pushed back hard ("stop doing build after build, figure out why first") — a fair correction. Mistakes file captures this anti-pattern.

### Round 5 — white avatar (textured-but-textureless)

Built, sideloaded, pressed B. **No more pink — avatar rendered as a solid white humanoid silhouette.** Major progress: the swap was happening, materials were being assigned, Sprites/Default was rendering. But the textures glTFast had presumably loaded weren't bound to anything — preservation code found `<NULL>` for every material's texture probe.

Conclusion: the textures never made it onto materials in the first place because glTFast had no PBR shader to bind them against in the build. We were chasing a downstream symptom of the actual bug (PBR shader missing).

### Round 6 — the canonical fix (Resources/ folder placeholder Material)

Stopped the trial-and-error loop and did intensive web research per user's pushback. Found the official Unity glTFast docs' two canonical fixes:

1. **Resources/ folder placeholder Material approach** — create Material assets that reference the glTFast PBR shadergraphs, place them in any `Resources/` folder. Unity bundles all `Resources/*` assets and their shader dependencies into the build. Shader survives the strip. Simple, idempotent, ~5min one-time setup.
2. **ShaderVariantCollection workflow** — record actual shader variants used during Editor playmode, save as asset, add to "Preloaded Shaders" list. More precise but requires a playmode capture step.

Went with option 1. Steps:
- Created `Assets/HoloBorn/Resources/`
- Inside, created two Material assets: `glTFastPbrPlaceholder.mat` (shader = `Shader Graphs/glTF-pbrMetallicRoughness`) and `glTFastSpecGlossPlaceholder.mat` (shader = `Shader Graphs/glTF-pbrSpecularGlossiness`).
- Verified via filesystem grep: shader GUIDs in the .mat files match the .shadergraph.meta GUIDs in the glTFast package exactly (`b9d29df...` for pbr-MR, `9a07dad...` for pbr-SG).
- Disabled `forceURPLitFallback` on TestGlbLoader (the runtime swap would now nuke the textures glTFast had just bound — having both fixes active would defeat each other).
- Disabled `matchWearerHeight` (hardcode 1.7m to avoid head-derived scaling failing on sit-test).

## Decisions worth remembering

- **Resources/ placeholder Material is the canonical Unity-documented fix for runtime glTFast PBR on URP** — not a hack, an officially recommended approach. Two Materials (pbrMetallicRoughness + pbrSpecularGlossiness) cover all common TRELLIS outputs. Idempotent: stays in source, gets bundled in every build automatically.
- **`forceURPLitFallback` is now off and should stay off** as long as Resources/ placeholders are present. The runtime swap actively destroys glTFast's correctly-bound textures. Mutually exclusive fixes.
- **`matchWearerHeight` is off, fixed 1.7m** — sit-test gave 1.27m avatar (head_y=1.189m / 0.937 = 1.27m). Until we have reliable wearer-height calibration logic that handles sitting/leaning, fixed scale matches legacy proven behavior.
- **ScanController spine matches the recovered legacy doc's expected behavior** — A=revolve (validate-frame gate first), X=burst (no validate gate, per shipped APK behavior verified by user empirical memory). InputAction binding pattern via the New Input System's `performed` callback (not legacy `Input` class — that was the source of FrameSender's "input dies after first press" bug).
- **APK + symbols folders gitignored** — `HoloBornAPK_BackUpThisFolder_ButDontShipItWithYourGame/` and `HoloBornUnity_BurstDebugInformation_DoNotShip/` are 60+MB build artifacts that don't belong in git. The folder names literally say "DoNotShip."
- **Unity Build Settings → Development Build + Patch and Run** — flagged for tomorrow as the iteration-speed lever. Today's build cycle was 15-30min per rebuild because of full IL2CPP recompiles; with these flags toggled, incremental rebuilds drop to 30-60 sec.
