### Real root cause (identified, not fixed yet — parked for tomorrow)

Earlier in the same crashed log, a line we should have grep'd for in the first place:

```
LogRHI: OpenGL MajorVersion = 3, MinorVersion = 2,
ShaderPlatform = GLSL_ES3_1_ANDROID, FeatureLevel = ES3_1
```

**The APK is running on OpenGL ES3.1, not Vulkan.** UE 5.5 + Quest 3's official Meta-recommended path is **Vulkan**; ES3.1 is supposed to be the fallback, not the primary RHI. The missing shader (`FScreenPSsRGBSourceMipLevelArray`) is the **texture-array variant** used by Mobile Multi-View stereo rendering — and UE 5.5's ES3.1 cooker has known issues producing this specific permutation when MMV is enabled. The cooker's `ShouldCompilePermutation` check returns false for ES3.1, but the engine still tries to use the shader at runtime → fatal error at startup.

**The fix (parked for tomorrow morning):**

1. **Project Settings → Platforms → Android → Build:**
   - **Support Vulkan**: enable
   - **Support OpenGL ES3.1**: disable (force Vulkan-only so we don't accidentally fall back)
2. Re-package (~45–90 min cold cook on Intel Mac). Vulkan platform (`SF_VULKAN_ES31_ANDROID`) has a different permutation table that includes the array shader properly.
3. Re-sideload + relaunch. Crash should be gone if Vulkan compiles the array permutation.

**Fallback path if Vulkan still crashes:** disable Mobile Multi-View entirely. Removes the array-shader requirement at the cost of stereo perf. Get a working baseline, then re-enable MMV after baseline works.

## Decisions worth remembering

- **The on-device UE log is the only useful crash signal on Quest.** `adb logcat` is drowned in `parfait` memory-monitor noise, and native tombstones are permission-denied without root. The actual file at `/sdcard/Android/data/com.YourCompany.HelloCubeSpike/files/UnrealGame/HelloCubeSpike/HelloCubeSpike/Saved/Logs/HelloCubeSpike.log` captures everything UE prints, including full callstacks and prior init log lines. **Default debug recipe for any future Quest crash:** pull this file first, grep for "Fatal error" or "Critical error", THEN read the lines BEFORE the fatal for context (RHI choice, shader platform, settings).
- **Stale-cache hypothesis was wrong, and the FIRST log had enough info to skip the cache detour.** Two consecutive crashes — one with stale cache, one with fully clean cache — produced the IDENTICAL error. That's a config problem, not a cache problem. If we'd grepped the first log for `ShaderPlatform =` and seen `GLSL_ES3_1_ANDROID`, we could have pivoted directly to the Vulkan/RHI angle. Instead burned 30+ minutes on a clean re-cook based on a guess.
- **UE 5.5 + Quest 3 wants Vulkan, not ES3.1.** Meta's official UE5 setup guide has been saying this. Yesterday's project setup defaulted to whatever UE5 auto-checks at project creation, which apparently included ES3.1 (and possibly nothing for Vulkan). Worth verifying explicitly for any future UE+Quest project — never trust auto-defaults for the RHI choice.
- **`Binaries/` is the magic folder for "didn't ask me to rebuild modules" behavior.** UE5 only prompts to rebuild C++ modules when `Binaries/` is older than `Source/`. As long as you don't delete `Binaries/`, the editor opens directly. This is desirable when troubleshooting cook issues — keeps the C++ side untouched while you wipe shader/cook caches.
- **The `.ini` files are authoritative for setting state, not the diary.** Yesterday's diary recorded "Instanced Stereo: ON" — today's screenshot showed it unchecked. Diaries are written by tired humans (or tired Claudes) at end-of-day; they're emotional + narrative artifacts, not source-of-truth. When debugging, read `Config/DefaultEngine.ini` directly.
