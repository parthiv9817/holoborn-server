### Burst-average code path validated end-to-end

Wrote `test_burst_average.py` against the 5 real Quest frames yesterday's `quest_test_mode` had captured at `results/quest_test_uploads/multiview_20260504_070012_572dbcad/`. First run output looked too blueish — suspicious. Production check: `app/routes/generation.py` line 120-130 wraps `cv2.imdecode → burst_average → cv2.imencode`, which is BGR throughout. The bug was in MY test script — I'd added a `cv2.cvtColor(averaged, cv2.COLOR_RGB2BGR)` thinking averaged was RGB. It's BGR. Channel swap created a Smurf. Fixed the test script, not production.

After the fix:
- per-pixel stddev across 5 frames: **4.149**
- MAE between averaged frame and frame_0: **4.949** (very close to theoretical noise-only floor of stddev/√5 ≈ 1.85)

Empirical conclusion: burst-averaging is doing what it's supposed to — modest noise reduction with minimal motion contamination. The 200ms burst window is too short for meaningful Quest-side movement to bleed in. Production code is correct, untested gap closed.

### Portraitizer's actual function nailed down

Ran `test_runpod_manual.py` with the burst-averaged frame as RunPod input (skipping the broken-billed OpenAI portraitizer entirely):

```
Job submitted: 3a61e2a0-6811-4b69-a6b4-15de0be186eb-e1
Job COMPLETED in 443.5s (cold start + GPU compute + queue)
GLB: 31,609,108 bytes (~31.6 MB)
GPU compute: 176s, magic bytes verified b'glTF'
Cost: ~$0.55
```

Opened in Babylon.js viewer. Anatomy correct, full body, lanyard preserved, segmentation clean. **But materials are visibly degraded vs yesterday's portrait-input GLB** — checked shirt collapsed to a dark blob, no specular separation on fabric, overall muddy/dim.

The empirical insight: the portraitizer's actual job isn't "input cleanup" — it's **lighting transformation**. Converting dim ambient Quest captures into studio-lit PBR-friendly inputs gives TRELLIS the contrast cues it needs for crisp materials. Without it, geometry is fine but textures are muddy.

This *empirically confirms* the OpenAI credit ask isn't vanity polish — it's load-bearing for visual quality. Stronger founder-update line than yesterday's draft.

## Decisions worth remembering

- **Mirror everything strategy** — when rebuilding from a known-shipped artifact, mirror the version pins, not "latest of each thing." Forensics on the deployed APK gave us Unity 6000.4.0f1, Meta XR Core 201.0.0, glTFast (specific name), permissions exactly. Reduces variables → reduces "why doesn't it work" later.
- **Unity 6000.4.5f1 acceptable patch drift** vs shipped's 6000.4.0f1 — same Unity 6.4 LTS minor, 5 patches newer (bug fixes only, no API breaks per Unity LTS semver policy). Risk of bite ~5%. Documented in README so we can blame it explicitly if anything weird shows up later.
- **Meta XR All-in-One SDK is split** — newer Meta releases ship separate Core SDK + MR Utility Kit + Interaction SDK + Voice SDK + Platform packages. Core SDK 201.0.0 + MRUK is the bundle for our use case (passthrough + camera access + scene + anchors).
- **`com.unity.cloud.gltfast`** is the modern name (Unity acquired Atteneder's package, rebranded). Still 100% compatible with the deployed APK's glTFast usage.
- **Building Blocks panel beats hand-wired components** — Meta's Project Setup Tool auto-applies 4 critical fixes when the Passthrough Camera Access block lands (manifest permission, splash screen, OVR Manager, camera background). Manual scene assembly skips those.
- **Public Unity repo** — parity with `holoborn-server` and `holoborn-gpu`, no secrets, green-square contribution graph.
- **Push to remote BEFORE feature code** — lesson written in blood, applied. The repo exists with empty scaffold + 4 recovered scripts. Tomorrow's ScanController work commits onto a real remote, not into a single-Mac death trap.
- **Defer build to tomorrow morning** when it's a 30-90 min cold cook and you log off at 7pm — first action of next day, Mac waits, you don't.

## Lessons written in blood

- **Verify color-space conventions before debugging "production is wrong"** — when my burst-average test output looked Smurf-blue, my first instinct was to suspect production code. Reading production carefully showed it was BGR throughout (cv2 default). The bug was in MY test script's added `cvtColor(RGB2BGR)` on already-BGR data. **Cost of careful read of production code: 60 seconds. Cost of debugging "production must be wrong": much more.**
- **External docs (CLAUDE.md) lose to user empirical memory of running the actual app** — CLAUDE.md said A=burst, X=validate-then-revolve. The deployed APK on the headset behaves: A=revolve, X=burst. The user's memory was the right ground truth. **For input bindings of a shipped app, ask the user/run the app — don't trust the spec doc.**
- **Unity 6 OpenXR renamed "Oculus Touch" → "Meta Quest Touch Plus"** — the legacy interaction profile name from older Meta XR SDKs no longer exists. Quest 3 controllers register as "Touch Plus" in modern OpenXR. Don't assume API names from older recovered code carry forward verbatim.
- **The "Project Setup Tool" auto-applied fixes are non-trivial** — when the Passthrough Camera Access block dropped, 4 manifest/permission/splash fixes were auto-applied. Without that block, we'd have had to manually edit AndroidManifest.xml + OVR Manager + splash settings. Use Building Blocks; don't manually wire what Meta's tools handle.
- **"Console error" vs "Console warning" matters when the user asks** — I called the IMGUI errors "warnings" twice. They were actually red errors. Even though they were Editor-only and didn't affect builds, the user was right to push back on the characterization. **Be precise about severity classification. Substantive non-impact ≠ severity downgrade.**
- **The `+ Install package by name` path is more reliable than searching Unity Registry view** — Unity 6's registry default view filters out some packages. For known package names (`com.unity.cloud.gltfast`, `com.unity.nuget.newtonsoft-json`), the explicit name install bypasses the filter and just works.
