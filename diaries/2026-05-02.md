### Build #2 — Failed at Gradle (manifest XML error)

Result: `BUILD FAILED in 40s` at `processDebugResources`:
```

```

Cause: Both our manual `+ExtraApplicationSettings` AND UE's auto-generated `ExtraApplicationSettings` (when `bPackageForMetaQuest=True` is on) put `<uses-feature>` inside `<application>`, but Android requires `<uses-feature>` at `<manifest>` top level, not inside `<application>`. The `<meta-data>` tags were fine.

Fix: Removed all our `+ExtraApplicationSettings` lines and edited UE's auto-generated line to keep ONLY the meta-data portion (Quest supportedDevices). Final state:

## The wall (root cause, verified)

**`FScreenPSsRGBSourceMipLevelArray` is a global engine shader defined in `Engine/Source/Runtime/Engine/Public/ScreenRendering.h`.** Used by stereo screen-pass when rendering from a Texture2DArray source.

In **UE 5.5**, the shader's `ShouldCompilePermutation` predicate has a regression for `VULKAN_ES31_ANDROID` shader platform — returns false (skip compile) even when the runtime needs it. The cooker silently skips, runtime VerifyGlobalShaders crashes.

Verified via web research:
- Cooker produces 1436 Vulkan shaders, missing only this one permutation
- Runtime requests it whenever stereo-array rendering is active
- Other devs hit it on UE 5.5.4 + Meta XR (Meta forum thread on Multi-View landscape flickering, same engine version, same plugin combo)
- Even **Oculus Fork of UE 5.5.4 doesn't fix this** (per Meta forum thread where dev on Oculus Fork hits related Multi-View bugs)
- No `.ini` setting overrides engine-level `ShouldCompilePermutation`
- No `r.SkipMissingGlobalShaders` or equivalent CVar exists in UE5
- No 5.5.5 hotfix release notes mention the fix

## The .ini split discovery (2026-05-02 ~16:25)

The reason 8 builds all crashed with the SAME shader signature regardless of UI toggling: there are TWO files setting `vr.InstancedStereo`, and the Mac editor and Android cook read different files.

```
Config/DefaultEngine.ini:146:    vr.InstancedStereo=True       ← what the Android APK gets cooked with
Config/DefaultEngine.ini:148:    vr.MobileMultiView=False
Config/Mac/MacEngine.ini:9:      vr.InstancedStereo=False      ← what the Mac editor UI shows
Config/Mac/MacEngine.ini:10:     vr.MobileMultiView=False
```

`Mac/MacEngine.ini` was created on 2026-04-29 13:51 to prevent the Mac editor from crashing on Metal RHI when previewing VR features (Meta XR doesn't ship Mac runtime modules, so VR Preview on Mac would crash without these overrides).

**Mac editor read order:** DefaultEngine.ini → then Mac/MacEngine.ini overlays on top → ISR=False shown in UI.
**Android cook read order:** DefaultEngine.ini → looks for Android/AndroidEngine.ini (none exists) → ISR=True ships in APK.

When the user toggled ISR in the editor UI on multiple builds today and yesterday, the editor showed it as already off (because of the Mac override) so the toggle either didn't write or the write target wasn't DefaultEngine.ini. Net effect: `vr.InstancedStereo=True` lived in DefaultEngine.ini line 146 unchallenged from project creation through Build #8.

**Rule for next time:** When debugging Android cook config on a Mac UE5 project, the editor UI is NOT the source of truth. `Config/DefaultEngine.ini` is. Verify with `grep` from terminal, not from Project Settings panel. Mac override files mask reality for any setting they touch.
