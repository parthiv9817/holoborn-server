## Q0 answered — the killer finding

Cloned `/tmp/holoborn-gpu/`, read `handler.py` + `run_inference.py` to understand the current single-image path. Then queried `microsoft/TRELLIS.2` via `gh api` for the pipeline source.

Found in `trellis2/pipelines/trellis2_image_to_3d.py`:

```python
def get_cond(self, image: Union[torch.Tensor, list[Image.Image]],
             resolution: int, include_neg_cond: bool = True) -> dict:
```

**TRELLIS.2-4B accepts a list of images at the conditioning layer.** The architecture supports multi-image natively. Top-level `pipeline.run()` only passes a single image, but `sample_sparse_structure`, `sample_shape_slat_cascade`, `decode_shape_slat`, `sample_tex_slat`, `decode_tex_slat`, `decode_latent` are all individually callable. We can compose our own `run_multi_image()` in the handler.

**This was the inflection moment.** Plan went from "uncertain feasibility, 2-3 weeks if we have to lift ComfyUI" to "feasible, 2-3 days handler work, no model downgrade." Founder timeline became confidently sayable.

Honest caveat to flag: "architecturally supported" ≠ "trained on multi-view." Model weights were trained primarily single-image. Quality lift is empirical — we'll know after the test runs.

## 5 instructive mistakes from today

### 1. Initial brainstorm scoped too broadly (9-section failure mode catalog)

**What I did:** Proposed a head-to-toe failure mode catalog with 9 sub-sections (hair / skin / eyes / beard / hands / clothing / body / anim / render) as the doc structure. Parthiv approved scope. Then the audit converged to "hair = root cause, multi-view = right fix" within ~30 min.

**What was wrong:** The 9-section structure was over-engineered relative to what the evidence demanded. Should have proposed a tighter scope (~3-4 sections) once the hair-cap dominance was visible in the Babylon screenshots. Could have saved 20-30 min of session overhead on a structure we ended up not using.

**Cost:** Brainstorming-skill time, not direct work cost. But it diluted the urgency of the actual finding.

### 2. Recommended TRELLIS 1.x downgrade as initial Path 1 before checking TRELLIS.2 source directly

**What I did:** Initial WebSearch surfaced TRELLIS-image 1.x's `run_multi_image()` as the "clean Python API path" — flagged as Path 1 with caveat that quality might regress.

**What was wrong:** Should have gone straight to the TRELLIS.2 GitHub source first (via `gh api`) instead of inferring from external docs. The `get_cond` signature was sitting there the whole time. ~15 min of WebSearch / WebFetch on TRELLIS 1.x docs could have been a single `gh api` call to inspect the .2 pipeline source.

**Cost:** ~15-20 min context detour + initial brainstorm pessimism (path 1 underwhelming, path 2 uncertain) that resolved when Q0 was actually answered.

**Memory worth saving:** for vendor SDK / model architecture questions, source code on GitHub is always step 1, docs/marketing/community posts step 2. (Already partially captured in `feedback_websearch_before_ai_opinions` but extension to source-first is worth its own entry.)

### 3. Started uvicorn + ngrok without checking bandwidth state of the current account

**What I did:** Parthiv asked for uvicorn + ngrok. Configured authtoken, started both, verified port 8000 alive → tried `/health` through the tunnel → bandwidth cap error.

**What was wrong:** Should have queried ngrok's bandwidth state (via API or dashboard URL) BEFORE running the tunnel and reporting "ready." 30-second check would have caught the cap and surfaced the blocker to Parthiv before he wasted Quest cycles on the assumption it was working.

**Cost:** ~5 min of false-positive "tunnel up" state. Recovered fast because curl `/health` immediately revealed the error HTML.

### 4. Recommended $10/mo paid ngrok escalation when Parthiv just wanted account #4 swap

**What I did:** When bandwidth blocker hit, I led with "upgrade ngrok to paid Personal tier $10/mo — Vipin already approved Meshy Pro $20/mo, this is smaller money for a recurring pattern."

**What was wrong:** Parthiv had already burned 3 free accounts and was in motion to swap to a 4th when I framed the paid upgrade as the "right" answer. Memory `feedback_no_timeline_pressure_framing` and brick-by-brick mode both apply — when user is in execution mode and a known-good workaround exists, don't introduce a vendor-billing escalation as the first recommendation. Save it for when the user himself asks "is there a better way."

**Cost:** Negligible (Parthiv made the call quickly anyway), but the wrong-default could have caused a longer derail if he'd entertained the paid option.

### 5. Implicit assumption that "same chat across machines" was possible until Parthiv explicitly asked

**What I did:** When Parthiv asked about being in the chat with me on Alienware, my first instinct was to suggest installing Claude Code on Alienware and resuming the session. Had to walk back to "actually Claude Code doesn't have cross-machine conversation sync."

**What was wrong:** Should have surfaced this limitation proactively when we first discussed the Alienware pivot (an hour earlier in the session). Letting Parthiv ASK the question added cognitive load — he was already in AAHHHH mode about the scope, and discovering a tooling limitation mid-conversation didn't help.

**Cost:** Minor — but the right move is to anticipate user expectations about tool affordances and clarify them ahead of time.

**Memory worth saving:** when a session involves a planned context handoff (machine switch, agent handoff, etc.), surface the tooling constraints early so the user can plan around them. Don't wait to be asked.

### Pattern across the 5

Three of these (#1, #2, #5) share a root: **acting on initial framing before testing it.** Initial scope frame (9-section), initial research framing (TRELLIS 1.x as Path 1), initial assumption about chat continuity — all defaults that an earlier check would have updated. Discipline for future me: when forming a framing in the first 5 min of a thread, treat it as a hypothesis to disconfirm, not a foundation to build on.

#3 and #4 are operational: small checks that prevent small errors, and matching the user's energy level when suggesting escalations. Both fixable with checklist-style discipline.
