## Morning — auto-rigging hypothesis from a dead-end integration

Walked in around 9:30 with the previous-day expectation that auto-rigging was "wire Meshy after demo polish." Vipin's "alive avatars" ask Tuesday had landed Meshy as the chosen path; today was supposed to be polish, not architecture.

Tested Meshy's web UI on the existing static GLB from yesterday (`manual_test_20260507_055734.glb`, 28 MB) — auto-rigged it, ran "Skill 1" animation. **Avatar deformed catastrophically — pants stretched outward into wing-like cloth shapes, lower-body topology behaved like a saree draping, silhouette collapsed during animation.** Wing-pants. Classic skin-weights-bleeding-across-fused-leg-topology failure. Bones were correct; the mesh was wrong.

**The hypothesis came from a different model's failure mode.** Few weeks back tried integrating LHM++ — different image-to-3D system that outputs Gaussian splats projected onto SMPLX. Couldn't get GLB extraction working due to texture projection + everything; integration failed. But during debugging, noticed LHM internally generated **30 different T-pose frames** as preprocessing before reconstruction. Stored that as "huh, why T-pose specifically" but didn't act on it.

This morning, looking at Meshy's wing-pants failure, the breadcrumb fired: *image silhouette → mesh topology → rig quality is a tight causal chain. When legs touch in the source, silhouette is one blob, TRELLIS can't separate the volumes, fused-leg GLB → auto-rigger bleeds skin weights across what should be independent body parts.* LHM was preprocessing INTO T-pose because that's what the architecture needs.

Spawned two parallel research agents:
1. OpenAI's official prompting docs for `gpt-image-1.5` (canonical structure, identity preservation, pose-modification guidance)
2. Community/forum empirical findings (Reddit, OpenAI Community Forum, fal.ai blog, Charlie Hills, Threads/Nano Banana posts)

Reports came back tight. Both converged: **upstream pose conditioning is industry standard. Every shipping avatar product (Wolf3D, Ready Player Me, RODIN, Avatar SDK, MetaHuman) does pose-normalization preprocessing — they just hide it behind UX.** The community-research agent specifically found CharacterGen (SIGGRAPH'24) and PoseMaster papers that make the same point in formal academic terms. We'd reverse-engineered a known industry pattern from a single failed integration.

Saved the architectural rule as project memory: `project_canonical_pose_required.md`.

## Late-morning — V3 prompt drafted + bench-tested via ChatGPT web UI

V3 prompt design synthesized from both research streams:

- Per-limb physical-state prose (not pose-name) — "Both arms extended approximately 30 to 40 degrees away from the torso" works; "stand in A-pose" fails per community findings

## Evening — actual Quest VR test session (the brick within the brick)

Up to this point, all Meshy validation was in their web UI. Decided to push through to Quest test before EOW: download rigged+animated GLB from Meshy, load via TestGlbLoader.cs, see it move in MR.

This was the second hardest hour of the project after yesterday's URP shader stripping debug. **Eight rounds of failure-fix cycles** before getting a clean render:

**Round 1 — frozen avatar.** glTFast 6.x default `AnimationMethod` is Mecanim (Animator without an AnimatorController → `Animator.Play(0)` is a no-op). Fixed by passing `ImportSettings { AnimationMethod = AnimationMethod.Legacy }` to `LoadGltfBinary`. First mistake on this round: I guessed `AnimationMethod` was on `InstantiationSettings`, got compile error, had to fix to `ImportSettings`. One build cycle wasted on a guess.

**Round 2 — wrong file served.** Inspector URL field was caching `/avatars/test.glb` (yesterday's static GLB). Even though I changed the script default, Inspector's serialized value wins. Fixed by `cp test_animated.glb test.glb` so the cached URL serves the right file.

**Round 3 — animation worked but textures looked "wet/oily/sparkly."** ChatGPT's first analysis (shared via web UI second opinion) flagged this as: backface culling on inverted normals (face polygons missing in Babylon), tangent space mismatch, mobile shader incompatibility. User confirmed Babylon viewer showed headless avatar.

**Round 4 — added `ForceDoubleSidedMaterials` walking all materials and setting `_Cull = 0`.** Quest face came back, no more disappearing polygons. But "wet/oily" sheen on shirt + jeans persisted.

**Round 5 — added `Mesh.RecalculateNormals + RecalculateTangents + RecalculateBounds` post-spawn on every SkinnedMeshRenderer.** Helped partially. Research agent later flagged this was probably making things worse, not better — RecalculateNormals computes normals from existing winding, can't fix inverted ones, just locks in the broken state.

**Round 6 — research agent dispatched** for thorough investigation. Returned with: most likely root cause is **tangent-basis mismatch revealed during skinning math** (skinning mutates tangents per-frame via Linear Blend Skinning, mismatched tangent space + skinned normal-map sampling = exact "wet/oily" pattern). Smoking gun: glTFast issue #437 — *"glTFast only computes tangents when required by a material."* Recommended: stop calling `Mesh.RecalculateTangents`, add `GLTFAST_SAFE` scripting define, set GPU Skinning to `GPU` not `GPU (Batched)`, do isolation test with rigged-only-no-animation GLB.

**Round 7 — isolation test conclusive.** Switched to `test_rigged_only.glb` (rigged, no animation tracks). **Static rigged GLB rendered cleanly. Same project, same shader setup, same Resources/ Materials.** Animation playback specifically triggered the artifact, not the mesh data itself.

**Round 8 — gentle animation hypothesis.** Per ChatGPT and research agent: aggressive animations (Boom Dance, Boxing Practice) create extreme joint deformation → maximum tangent stress → maximum visible artifact. Idle = breathing, slight sway = minimum deformation → should be clean. User re-exported from Meshy with **only Idle**. Loaded into Babylon → CLEAN. Loaded onto Quest → **CLEAN.** Hypothesis validated. Captured 28-second screen recording at 17:43 — full PBR avatar idling in MR passthrough, your actual office in the background. The win.

## The architectural insight that emerged from the failure cycles

Tested Boxing Practice as a single-clip GLB in Babylon → **also clean.** Re-tested with merged 4-clip GLB → wet/oily artifacts back. **The bug is Meshy's `Single file ON` multi-clip merge step**, not animation amplitude, not polycount, not URP, not skinning math. Single-clip GLBs render cleanly with ANY animation.

Saved as project memory: `project_meshy_multiclip_merge_bug.md`. Workaround for future multi-clip needs: do Mac-side merging via `gltf-transform` Python tool, bypass Meshy's broken merge entirely.

This rules out earlier hypotheses (animation amplitude, polycount, tangent space, GPU skinning) as primary causes. Ruled out a lot of expensive exploration paths in one finding.

## Decisions worth remembering

- **Canonical pose conditioning is non-negotiable** for HoloBorn auto-rigging quality. Don't propose Meshy/TRELLIS config tweaks as fixes for wing-pants symptoms — root cause is always upstream silhouette ambiguity. Saved as project memory.
- **Single-clip GLB exports from Meshy are the production pattern.** Meshy's `Single file ON` merge produces corrupted multi-clip GLBs (wet/oily/sparkly artifacts during animation). For multi-clip needs, do Mac-side merge with `gltf-transform`. Saved as project memory.
- **glTFast 6.x default `AnimationMethod` is Mecanim**, which adds an Animator without an AnimatorController → `Animator.Play(0)` is a no-op → frozen avatar. Force Legacy via `ImportSettings { AnimationMethod = AnimationMethod.Legacy }` for our use case (single-clip GLB, simple Play()).
- **Force `_Cull = 0` on every material post-spawn.** Meshy's GLB exports ship with inconsistent triangle winding / inverted normals on parts of the mesh (notably the head). The glTF `doubleSided` flag isn't set, so Babylon and Quest URP both backface-cull the wrong-facing polys.
- **`forceURPLitFallback` should stay OFF** as long as Resources/ placeholder Materials are present. The runtime swap was a yesterday-fix that's no longer needed; the placeholder Materials handle URP shader stripping correctly.
- **Past-Parthiv's GPU CI/CD pipeline saved the day.** Without GitHub Actions auto-building the Docker image on push, tuning TRELLIS sampler params would have required cloning + rebuilding + pushing the image manually from this Intel Mac (no local GPU). One-time infrastructure investment from a previous machine paid back today and will keep paying.
- **Image tag bumping (v3 → v4) is the production-safety pattern.** RunPod can pin to `:v3` for rollback if `:latest` breaks. Costs nothing, gives a parachute.
- **`demo_max` preset (200k decimation, 22 steps, fal.ai range endpoints) is the production default.** Same visible quality as `demo_premium`, 24% smaller file, faster Quest download. Locked.
- **The static-vs-animated differential test is THE diagnostic for skinned-mesh rendering bugs.** If static rigged GLB renders cleanly but animated GLB doesn't, mesh data is healthy and the issue is animation playback / skinning math / multi-clip merge. Narrows the search dramatically.
- **Meshy multi-clip merge is broken; gentle animations work either way; aggressive animations work as single-clip.** Don't conflate "aggressive animation" with "Meshy bug" — the bug was the merge step, not the animation amplitude.

## Lessons written in blood

- **Visual interpretation of Quest screenshots is a weak skill of mine right now.** I called the avatar "looks great!" multiple times when it had visible artifacts the user could see. Same anti-pattern as 2026-05-06 ("iterate before researching"). Especially on Quest where build cycles cost real time, the threshold for "stop, research, then iterate" is way lower than it feels.
- **Two-pass V2→V3 portraitizer was a wrong steer.** User immediately spotted the cost/latency/dependency multiplication problem. I'd put two-pass on equal footing with single-pass without weighing it. Should have led with "improve the input via free local preprocessing" instead of "add another paid API call."
- **Conflated input quality with model failure.** Initially read V3's face-drift result as "V3 is broken." User correctly pushed back: input was degraded (dim Quest passthrough, occluded face) — garbage-in-garbage-out is part of the failure attribution, not all of it. **Treat the user's empirical assessment of input quality as ground truth before assigning blame to the model.**
- **Wrong API hypothesis with `InstantiationSettings.AnimationMethod`.** Guessed it was on `InstantiationSettings`, was actually on `ImportSettings`. Compile error caught it but cost a build cycle. Lesson: when working with an API I don't have memorized, grep the actual source (`Library/PackageCache/com.unity.cloud.gltfast@.../`) BEFORE writing code that depends on the guess.
- **Started troubleshooting backend serving when ngrok was the actual blocker.** When `curl` returned HTTP 403 with HTML body, didn't immediately read the HTML — would have surfaced `ERR_NGROK_725` (bandwidth limit) instantly.
