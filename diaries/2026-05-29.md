## Decisions worth remembering
- **Placement: post-retex / pre-rigging.** Validated path; cleanest code insertion (one spot); graft stays consistent (graft source = decimated retex = same UVs as rigged mesh). Pre-retex is theoretically cleaner (Meshy would bake UVs for the 81k mesh) but untested + needs RunPod to validate → parked as a future optimization.
- **`-noq` is mandatory.** Default quantization scrambles UVs on glTFast 6.18. The `-si` does the tri reduction regardless, so no downside to `-noq`. Bonus: plain glTF is safest for Meshy rigging input.
- **KTX2 / meshopt are OFF the table.** Unity project (glTFast 6.18.0) has NO KtxUnity and NO meshopt-decompress package — `-tc`/`-c` would silently fail to load. Geometry-only `-si` is what's supported. KTX2 (to shrink the ~30MB of 4K textures) needs KtxUnity added + an APK rebuild — separate decision.
- **Flag default OFF** until a full RunPod e2e confirms the front-end flows in. Flip ON + merge after that.
- **Tunnel: stay on ngrok through dev.** Evaluated Cloudflare Tunnel + Tailscale Funnel to escape ngrok's bandwidth-cap churn. Conclusion: free Cloudflare quick-tunnel still gives a random URL per restart → still needs APK rebuilds → migration is pain-for-no-payoff on a non-deployed POC. Revisit (cheap domain, or Tailscale Funnel) only when Vipin greenlights deployment.

## The locked recipe
```
gltfpack -i <retex.glb> -o <out.glb> -si 0.18 -noq
```
Runs on the **post-retex / pre-rig** GLB. ~452k → ~81k tris. To enable in production: `DECIMATE_BEFORE_RIGGING=true` + **`GLTFPACK_BIN=/usr/local/bin/gltfpack`** in `.env` (so the uvicorn process finds it regardless of PATH), restart uvicorn, run one e2e, re-measure `riggedBoundsToFeetOffset`, then default ON + merge.
