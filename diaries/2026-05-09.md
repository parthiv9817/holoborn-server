## Mid-afternoon — first Quest tests + vortex bug + lerp bug

**First test on Quest:** vortex appeared as a "single ring rising as a unit" instead of a fountain. Parthiv flagged it. Diagnosed: my vortex math correlated orbit angle and vertical drift linearly with cube index → perfect helix shape. Fix: per-cube random vertical phase offset (deterministic seed) decouples horizontal angle from vertical position. Cubes now scatter throughout the column at all heights = continuous fountain.

**Second test on Quest:** post-vortex, all cubes collapsed to a single point on the floor. Parthiv pushed back hard against my "you're looking the wrong way" theory: *"i have eyes you think i am lying lol."* He was right, I was wrong. Diagnosed: cubes had stale lerp state from spawn (`targetPos = spawnOrigin`, `lerpDuration = 0.0001`). UpdateVortex was directly writing positions but not syncing the cube's lerp state. When UpdateVortex stopped firing (P2a → P2b transition), `CubeCloudManager.Tick` kept lerping to the stale spawn target → all cubes converged to that single point. **Fix:** sync `cube.startPos = newPos; cube.targetPos = newPos; cube.lerpStartTime = now;` inside UpdateVortex so Tick becomes a no-op. **Lesson:** when user says they see X and logs disagree, eyes win — logs lie via stale buffers, but eyes don't lie about device behavior.

**Cube count + radius tuning:**
- 300 → felt sparse → 1000 → still sparse but starting to read
- Cross-AI Brief #1 (cube count): both ChatGPT and Gemini converged on 0.5m radius (not 0.75), 2000 cubes, 0.03 size, no GPU instancing
- Applied: `spawnRadius = 0.5f`, `initialCubeCount = 2000`, prefab scale 0.02 → 0.03, `Cast Shadows = Off`
- 2000 cubes felt cluttered → reduced to 1500 → silhouette still clumpy

**Silhouette clumping issue:** cubes formed discrete blob clusters at body landmarks (head, chest, hip) instead of evenly distributing. Diagnosed: Y Bot mesh has joint-concentrated vertex density. Vertex sampling biases toward those high-density regions.

Cross-AI Brief #2 (silhouette uniform sampling): both AIs converged on Heitz 2019 area-weighted triangle sampling + barycentric random points + Y-axis anatomical density bias.

Implemented `SilhouetteBehavior.SampleSurfaceUniform()` — pre-computes cumulative triangle area, picks triangles weighted by area, generates uniform barycentric points, applies Y-axis lower-body discard at 30% rate.

## Decisions worth remembering

- **The plastic look is industry-wide for single-photo-to-3D AI**, not TRELLIS-specific. ChatGPT's pushback: "set expectations for stylized digital twin, not AAA metahuman." TRELLIS bakes shadows + lighting into BaseColor because trained on 2D renders. No factor multiplier or roughness fix can fully remove that. Set founder expectations accordingly.
- **HDRI cubemap on Reflection Probe is the dominant PBR fix on Quest MR.** Without one, probe captures empty Unity scene → black void → PBR specular has nothing to reflect → uniform glossy plastic. Custom probe Type + Poly Haven studio HDRI cubemap fixes 60-70% per cross-AI consensus.
- **glTFast 6.x URP Shader Graph property names are camelCase no-underscore.** `roughnessFactor`, `metallicFactor`, `baseColorFactor` — NOT `_RoughnessFactor` etc. This is verified at glTFast source code (`Runtime/Scripts/Material/MaterialProperty.cs`). Different from URP/Lit convention.
- **Lerp state must sync with direct position writes.** Any code path that writes `transform.position` directly (bypassing `Cube.SetTarget`) MUST also sync `startPos`, `targetPos`, `lerpStartTime` — otherwise `CubeCloudManager.Tick` keeps lerping toward stale targets and pulls cubes back to bad positions when the direct-write loop stops.
- **Vortex math: orbit phase and vertical phase MUST be independent random.** Linear correlation between them creates a helix (looks like a dragon). Random per-cube vertical offset (deterministic seed) → continuous fountain.
- **Y Bot mesh is vertex-density-biased toward joints.** Vertex sampling clumps cubes at head/hands/feet. Heitz 2019 area-weighted triangle sampling + barycentric random points fixes this — gives uniform surface distribution regardless of mesh detail concentration.
- **`os.environ` is NOT the same namespace as pydantic-settings `.env` loading.** Adding env vars to `.env` populates `Settings.<field>` but NOT `os.environ`. Use `from app.config import settings; settings.<field>`.
- **Defensive Inspector value correction at runtime saves rebuilds.** ScanController's `serverUrl` Inspector value persisted as `https://.../avatars/test.glb ` (with trailing space) across rebuilds. Adding runtime check for malformed values (`Contains("/avatars/")`, `!= Trim()`) auto-corrects to PERM_URL on every Start() — so Inspector serialization can never bite again.
- **Unity asmdef references are one-way.** `HoloBorn.SpawnRitual` (asmdef) cannot reference `Assembly-CSharp` (default). So files in SpawnRitual that need to call into Assembly-CSharp classes must EITHER move out of the asmdef OR inline private copies of the dependencies.
- **Trust the user's eye over the logs.** Multiple times today I tried to explain visible behavior away with log analysis. User was right every time. Logs lie via stale buffers, USB disconnects, missing process IDs. Eyes don't lie about what's on screen.
