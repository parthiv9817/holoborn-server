# 2026-05-05 — Mistakes

## 1. BGR/RGB color swap in `test_burst_average.py`, then almost suspected production

**What happened:**
Wrote a standalone test script for the production `burst_average()` numpy code path. First run output (`results/burst_test/averaged.jpg`) came out visibly blueish — skin tones inverted, clothes too cool. My instinct was to debug whether `burst_average()` was producing wrong color data. Spent ~2 minutes opening `app/services/preprocessing.py` and `app/routes/generation.py` looking for production bugs. Production was correct. The bug was in my test script: I'd added `cv2.cvtColor(averaged, cv2.COLOR_RGB2BGR)` before `cv2.imwrite()`, assuming `averaged` was RGB. It's BGR (cv2.imdecode default). The conversion swapped R and B channels into the wrong order before write.

Production verified clean (line 121 + 127 in `generation.py` keep BGR end-to-end). Fixed my test script to write `averaged` directly without conversion. Re-ran: stddev 4.149, MAE 4.949 (was 21.578 with the swap, because of the off-by-channel difference inflating the diff metric).

**Why it happened:**
- Wrote the test script quickly without checking what color space `decode_jpeg` returns. Assumed RGB because most modern image APIs use RGB; cv2 uses BGR for legacy reasons.
- Didn't run a quick sanity dump of an array slice to verify channel ordering before writing the conversion call.
- When the output looked wrong, my first hypothesis was "production must be wrong" rather than "my new test script must be wrong" — the older code is more likely correct than the brand-new code I just wrote.
- Cost: ~5 minutes of confused debugging + a Smurf-blue screenshot the user had to question before I caught my own bug.

**Rule for next time:**
**Before writing color-conversion calls, verify what color space the input function returns.** For cv2-based pipelines: `cv2.imdecode` returns BGR, `cv2.imread` returns BGR, `cv2.imwrite` expects BGR, `cv2.imencode` expects BGR. No conversion needed within a cv2-only pipeline. RGB conversions are only needed when crossing to PIL, OpenCV-PIL boundaries, or display formats.

More general rule: **when brand-new test code produces unexpected output, suspect the test code first — not the older production code that has been working.** Production code has at minimum been compiled, possibly tested, possibly shipped. Test code I wrote 30 seconds ago has none of those guarantees.

## 2. Recommended input mapping from CLAUDE.md without verifying against the deployed APK

**What happened:**
After the web chat returned a partial recovered ScanController with input mapping `A=revolve, X=single`, the web chat itself flagged that "shipped APK per CLAUDE.md uses A=burst, X=validate-then-revolve" — i.e. the recovered code is supposedly inverted. I copy-pasted that conclusion directly into my response: "Input mapping is INVERTED in recovered ScanController — Recovered says A=revolve, X=single. Shipped says A=burst, X=validate. Don't carry recovered mapping forward; rebuild per CLAUDE.md."

User pushed back from memory of using the deployed app: *"i did it tbh A for revolving around the user multi image input X for burst 5 image bust capture that's all brother."*

So the actual shipped behavior is **A=revolve, X=burst** — the recovered ScanController's A-button logic is correct (keep it), but it's missing the X-button burst path entirely (burst was added on the broken Mac after April 28). CLAUDE.md is the document that's wrong about input mapping.

If user hadn't caught it, we would have wired A-button to burst capture and X-button to validate — opposite of what the demo on Quest currently does. Founder watching the demo would have pressed A expecting revolve, gotten burst instead, asked "what changed?" — that would have been a confidence-eroding moment.

**Why it happened:**
- I trusted the chain: CLAUDE.md → web chat citing CLAUDE.md → me re-citing the web chat's citation. Three hops, each treating the previous as authoritative without verifying against the actual deployed app.
- CLAUDE.md is a *spec doc*, not a *behavioral truth*. The deployed APK is the behavioral truth. CLAUDE.md says A is on left controller (which is physically wrong — A is on right controller of Quest 3); the input mapping section should be assumed unreliable based on that physical-layout error alone.
- The user has been using the deployed app for weeks. Their muscle memory of "I press A to spin around, X to capture in place" is the highest-fidelity ground truth available short of decompiling and reading the APK's IL2CPP-recovered code.
- I dropped a critical claim ("shipped is A=burst") into a recommendation without flagging the verification gap. Should have framed it as "per CLAUDE.md, this should be the mapping — verify against your memory of using the app before we wire it."

**Rule for next time:**
**For any claim about shipped-app behavior — input bindings, UI flows, default values — treat the user's memory of using the app as higher-fidelity than spec docs.** Spec docs drift. Deployed apps don't.

Concretely: when about to recommend wiring input/UI behavior based on a doc, first ask: *"does this match what you actually remember the app doing?"* The user has the live evidence; the doc has the historical intent. They diverge.

Connected: this is the same anti-pattern as 2026-05-04 mistake #4 ("Recommended UE 5.4.3 to user before web-verifying it was Mac-confirmed"). Both are "made a confident recommendation based on a chain of citations without going to the empirical ground truth." Today's verification path was the user's memory; yesterday's was a web search. Same lesson, second instance: always have an empirical verification step before confident recommendations on stack/behavior questions.

## 3. Called the IMGUI errors "warnings" twice without verifying severity icon

**What happened:**
After the user dragged components onto Inspector slots, the Console started showing "Please specify an image or a imageAspect" repeated messages from `UnityEngine.GUIUtility:ProcessEvent` (`Modules/IMGUI/GUIUtility.cs:224`).

I called them "warnings" without checking the icon color, then framed them as "harmless cosmetic Editor noise." User pushed back twice:
- *"but that's not a warning bruh"*
- *"view the screenshot carefully breh"*

On second look at the screenshot, the messages were rendered with red error styling — they were **errors**, not warnings.

I eventually conceded the severity but maintained the substantive claim that they were Editor-internal IMGUI noise (which was correct — Unity's `Modules/IMGUI/` is Editor-only code, doesn't ship in builds, file path `/Users/bokken/...` is Unity's CI build server). After a Unity restart the errors disappeared, confirming the cosmetic interpretation.

The substantive read was right; the severity classification was sloppy and I doubled down before checking.

**Why it happened:**
- Glanced at the Console output text without zooming in on the icon color. "GUIUtility:ProcessEvent" *sounded* like the kind of thing that's typically a warning in Unity ecosystem chatter, so I went with that.
- When user pushed back the first time, I hedged ("severity matters less than the source") rather than going to verify. Got pulled into defending a position I hadn't fully checked.
- On second pushback I finally went and looked carefully, eat hat.

**Rule for next time:**
**Before classifying severity (error vs warning vs info) of any Console message, verify by looking at the icon, not just the message text.** Unity Console icons:
- Red ⊘ circle = Error
- Yellow ⚠ triangle = Warning
- Blue ⓘ info icon = Info / Log

If the screenshot resolution doesn't make the icon clearly distinguishable, ask the user explicitly: "is the icon red, yellow, or blue?" — don't guess.

More importantly: **when a user pushes back on a classification, go re-verify before re-stating the position.** The hedge "severity matters less than the source" was technically true but read as "I'm not going to check, I'll just shift the argument." That's not respecting the user's signal.

The substantive non-impact (Editor-only, doesn't affect builds) was still correct, but I should have led with the corrected severity *and* the substantive read together: "you're right, those are errors not warnings. They're red errors, but they're from Unity Editor IMGUI internals and don't ship in builds." That's both accurate and respects the pushback.

## 4. Gave UI directions referencing an Anchor Presets popup the user couldn't find, instead of the simpler Width/Height path

**What happened:**
For Brick 3c (sizing the Camera Feed Display RawImage to fill the canvas), I told the user to use Unity's "Anchor Presets" popup — "Anchor Presets → stretch-stretch" with `Left/Top/Right/Bottom = 0`. The user got stuck because the Anchor Presets popup wasn't obvious in their Unity 6 view ("there are no anchor presets breh").

When I saw the next screenshot, I realized the simpler instruction was just *"set Width=800, Height=500 to match parent canvas dimensions"* — which doesn't require the popup at all. We didn't need stretch-anchoring; the canvas isn't being resized at runtime.

Wasted ~5 min of user time on a UI element that wasn't necessary for our use case.

**Why it happened:**
- Anchor Presets is the standard Unity UI workflow taught in tutorials, so it's what came to mind first when I thought "size a RawImage to fill its parent."
- Didn't think about the actual constraint: in our scene, the canvas Width/Height never changes at runtime. We don't need anchor-stretch for responsive behavior. We just need pixel dimensions to match.
- Defaulted to "the more general technique" when "the simpler exact-value entry" would do.

**Rule for next time:**
**For Unity scene wiring with junior-Unity users (or Unity-newcomer in our case, post-broken-Mac rebuild), prefer concrete absolute values over relative/anchor systems unless the dynamic behavior is needed.** Width=800, Height=500 with default centered anchors works for our static canvas. Anchor stretching only earns its complexity when the canvas resizes, which ours doesn't.

More general: **when explaining a UI step, ask "what's the simplest sequence of clicks that accomplishes this exact goal?" — not "what's the most elegant Unity-idiomatic way."** The user is solving the goal, not learning Unity best practices.

## 5. Recommended Unity Hub default download without anticipating the Apple-Silicon-vs-Intel mismatch

**What happened:**
Initially recommended "https://unity.com/download" as the install path for Unity Hub. User downloaded `Unity Hub 3.17.3-arm64.dmg` (Apple Silicon build) on their Intel Mac, macOS refused to install it (no-entry icon).

We then verified Mac is Intel via `uname -m` (`x86_64`) + `sysctl -n machdep.cpu.brand_string` ("Intel(R) Core(TM) i7-1068NG7"). Re-downloaded the Intel/universal `UnityHubSetup.dmg` (no `-arm64` suffix). Installed cleanly.

Cost: ~5 min downloading the wrong DMG + redownload time.

**Why it happened:**
- I knew this Mac is Intel (it's been the entire context of every diary entry since 2026-04-27 — Iris Plus iGPU, 16GB RAM, etc.). Should have proactively flagged the architecture mismatch in the install instructions.
- The 2026-05-04 mistakes file literally has an entry (#4) about this exact pattern: "Recommended UE 5.4.3 to user before web-verifying it was Mac-confirmed." Same anti-pattern. Recommended a download without ensuring it's the right architecture for this specific user's hardware.
- Defaulted to "go to the official download page" without specifying which architecture build, when I knew the architecture and could have given the explicit Intel URL upfront.

**Rule for next time:**
**For any installer download recommendation, specify the exact architecture build the user needs.** Apple Silicon and Intel Macs need different DMGs in 2026 for many tools (Unity Hub, JetBrains products, browsers, Office). Default browser-detection on the download page is unreliable — defaults to whichever the user's browser hints at, sometimes wrong.

For this user specifically (Intel i7-1068NG7 x86_64): always recommend the Intel/universal binary, not the arm64 build. This goes in muscle memory: the Mac is Intel.

Concretely: when about to write "go to https://example.com/download" — instead write "go to https://example.com/download → look for **Mac (Intel)** option" or give the direct Intel URL.

Same pattern as 2026-05-04 mistake #4. Pattern is establishing — for any vendor download/install recommendation:
1. Verify the exact OS + architecture the user has
2. Verify the exact tool's variant matrix exists (universal? Intel-only? arm64-only? per-OS DMGs?)
3. Recommend the specific variant, not the generic page
