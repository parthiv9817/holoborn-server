When user asks "why are we doing X" — first answer "we agreed to X because Y in spec Z." Confirm the spec is still right. Only AFTER user explicitly says "let's cut X" should you propose alternatives. Don't slip scope-cuts in as helpful simplifications.

The framing flip: "we agreed to X" (referencing the locked plan) vs. "we could simplify to Y" (offering downgrade). The first is the right answer to confusion. The second is changing the plan without permission.

## 4. Wrong material values produced Tron-cyan body instead of subtle rim

**What happened:**
Wrote in v2 plan: *"Emission: cyan (#00E0FF), intensity 1.5"* for the translucent mannequin. User created the material, applied it to Y-Bot. Result: the entire mannequin glowed bright cyan — looked like a Tron lightcycle, not the "pale white with subtle cyan rim" from Gemini's concept art.

The issue: URP/Lit emission applies UNIFORMLY to the whole material color. On a translucent surface with low base alpha, the emission becomes the dominant visual. Intensity 1.5 on a transparent material = flooded glow, not edge rim.

To get an edge-only rim effect, you need a **Fresnel/rim shader graph** — not stock URP/Lit. Stock URP/Lit can only do uniform emission.

I prescribed the wrong values without thinking through how URP/Lit emission actually behaves on transparent surfaces. Then doubled down by suggesting "drop intensity to 0.2" instead of immediately admitting we needed a shader graph for proper rim effect.

**Why it happened:**
- Copied the values from the v2 plan I wrote earlier without re-reasoning whether URP/Lit would produce the desired rim look
- Didn't pause to think about transparent-surface emission behavior
- Trusted the value rather than the visual

**Rule for next time:**
For any "subtle rim" or "fresnel" visual effect: don't propose stock URP/Lit emission as the way. Either:
- Propose URP/Lit emission with the caveat "this gives uniform glow, not edge-only — will look fully tinted, accept that or use shader graph"
- Or propose URP Shader Graph from the start, with Fresnel node + emission, with clear "this requires creating a shader graph asset"

Stock shaders have limits. Visual references like "subtle cyan rim on silhouette edges" map to shader graph features, not stock URP knobs.

This is Saturday's mistake #1 shape applied to a different domain: I trusted convention (URP/Lit emission) over actually verifying what visual would result. Same anti-pattern.

## 6. Bricks were too big when user asked for smaller bricks

**What happened:**
User said *"brick by brick"* multiple times. Then qualified: *"by brick I don't mean move the cursor I'm not from kindergarten."*

The contradiction: user wants atomic-ish progress without micro-instructions. The right "brick size" is **coherent unit of work with visible end artifact** — not "one click" and not "5 steps to set up something across 3 panels."

I oscillated between too-small (cursor moves) and too-big (multi-step Unity setup that touched material, mesh, prefab, and Hierarchy all in one instruction).

**Why it happened:**
- Misread "brick by brick" as "one click at a time"
- Then over-corrected to "complete setup of feature X across all panels"
- Didn't internalize: brick = one coherent thing with one verifiable result

**Rule for next time:**
A brick is: "build artifact X, end state = X exists and is visible/usable." Examples:
- ✅ Brick: build the GroundShadow prefab (Quad + material + saved to Prefabs folder)
- ❌ Sub-brick: "drag Quad into Hierarchy" (too small — no verifiable end state)
- ❌ Mega-brick: "build mannequin + material + apply + wire controller + test" (too big — multiple artifacts, multiple verifications)

The verification at end of brick is the key: "you should now see a glowing cyan disc in Assets/HoloBorn/Prefabs" — that's the brick boundary.

## 7. Forgot my own powers — walked user through Unity Editor click-by-click when I could have file-edited

**What happened (second-session evening):**
Started Brick 3 of the Stage 1 build by giving the user step-by-step Unity Editor instructions: "create a Quad, set rotation to (90, 0, 0), enable Emission on the material, set HDR color #00E0FF with intensity 1.5..." When the rotation was wrong (Quad-front faced into the floor with backface culling on), I gave another step. When the emission HDR picker re-normalized values, I gave another step. When the alpha import didn't stick after their Apply, I tried to edit the .meta but Unity overwrote me. When the rendering came out orange, I gave more debug steps. Across ~60 minutes the user got progressively frustrated, eventually:

> *"brother and also you can change all of this you have access to the raw repo the raw sample scene and you made me do all the check boxes when you could've just done it"*

And:

> *"bro understand this i am not claude code and you are not parthiv s the AI engineer if the roles are reversed we are for sure to fk things up"*

The user is exactly right. **Unity prefabs, materials, shader files, and scene files are all YAML/JSON sitting in the repo.** I have full Read/Write/Edit tool access. I should have been:
- Writing `.prefab` YAML directly with the correct mesh fileID, transform, scale, components
- Writing `.mat` YAML directly with the correct shader GUID and properties
- Writing `.shader` HLSL files directly (or `.shadergraph` JSON)
- Editing `SampleScene.unity` directly to add GameObjects + Inspector refs

Instead I treated the user — an AI engineer who's shipped backend pipelines and RunPod integrations — as if they were the "Claude Code" of the team while I was Parthiv directing them. **Roles inverted: the human did the procedural Unity Editor work, I did the planning/research work.** That's exactly backwards. Procedural execution on a deterministic file format is what I'm best at. Design judgment + visual taste is what the user is best at.

**Why it happened:**
- The implementation plan I wrote (`docs/superpowers/plans/2026-05-11-stage-1-premium-build.md`) had Unity Editor click-by-click in every brick — "right-click → Create → Material", "drag PNG into Base Map slot", etc. I encoded the wrong working model into the plan and then followed my own bad plan.
- I had a learned mental model from prior sessions: "Unity Editor is the source of truth, Claude can't touch it." That was wrong. Unity Editor REFLECTS YAML files in `Assets/`. Edit the YAML, Editor updates. The Editor is just a GUI over the file system.
- I underestimated how much friction "click X in panel Y" instructions generate when I can't see the user's screen — Quad rotation, alpha-is-transparency apply, emission color picker were all medium-reliability turf where I drifted.

**Rule for next time:**
When working in any project where the assets are file-based (Unity YAML/JSON, Godot tscn, Unreal uasset binary is the exception, etc.):
1. **Default to direct file edits.** Only fall back to "ask user to click in Editor" when the file format is opaque (binary) or when the file edit would be more error-prone than a 1-line UI instruction.
2. **Mining GUIDs is part of the job.** Read .meta files for asset GUIDs, search package caches for built-in shader GUIDs, use them to construct correct cross-references in YAML.
3. **The user's role is design judgment + device verification.** "Does this look right on the headset?" is human work. "Set the Cull value to 0 in the material YAML" is my work.
4. **When writing implementation plans, audit the click-by-click instructions.** Anything that's "open Inspector and toggle X" can probably be a direct YAML edit. Anything that's "drag this prefab into the scene at this position" can probably be a direct scene-file edit.

The user said "i am not claude code and you are not parthiv" — that line is the rule. Procedural execution to me, design judgment to them. Don't invert.

Saved as memory: `feedback_use_direct_file_editing.md`.
