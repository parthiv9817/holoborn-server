## 1. Attributed visual results to code that wasn't actually in the build

**What happened:**
Parthiv pulled Quest screenshots at 18:32 showing a dimmer avatar than the 18:15 set. I immediately diagnosed it as the result of my Round 3 IBL retune (TopColor 0.55) + ACES tonemap change being too aggressive, and applied a Round 4 correction.

Parthiv then clarified: *"the quest you just captured i didnt include the HDRI part nor the research agent's fix that's yet to be viewed just to point it out."*

The 18:32 screenshots were actually from a Round 2 build (TopColor 1.00, RefInt 0.30). Round 3 + ACES had never made it into a build yet. My diagnosis was attributed to code that wasn't running.

**Why it happened:**
- I assumed file edits → build → test was happening continuously
- Didn't ask "which build is this?" before reading the visual
- Pattern-matched "dimmer avatar" to "the last thing I changed reduced brightness"

**Rule for next time:**

Before attributing visual changes to recent code edits, **explicitly confirm which build is running on Quest**. Two ways:
- Ask the user: *"to confirm — was this build done after my last edit?"*
- Use logcat: the `[IBLBootstrap] applied — ... ambient=X.XX, reflection=Y.YY` line tells you exactly which values are live. Read that BEFORE diagnosing visual differences.

The bigger principle: **don't diagnose tested-config A using assumptions about config B.** If the build is unclear, find out. Two minutes of verification beats a misdirected Round 4 fix.

This is a flavor of `feedback_test_matrix_before_proposing.md` — by Round 3 I should have written a {build version, IBL config, tonemap, observed result} matrix and checked which row I was actually looking at. Didn't do it. Should have.

## 2. Built procedural cubemap before asking if real HDRI was available

**What happened:**
When I first identified Loophole #2 (no IBL on Quest) at ~17:00, I jumped straight to building `IBLBootstrap.cs` with a procedural studio-gradient cubemap. The procedural approach was a creative solve given I couldn't easily download an HDRI via my tools (WebFetch doesn't reliably handle binary `.hdr`/`.exr` files).

Then ~90 minutes of iteration later, the research-agent recommended the proper fix: download a real HDRI from polyhaven, import as Cubemap with Specular convolution. Parthiv could trivially do this manually. He did, and we got the proper Layer 5 fix in ~20 minutes including .meta tuning + Resources auto-load.

**Why it happened:**
- I scoped the fix to "what I can do alone with my tool set" instead of "what's the best fix, even if a 30-second human action is needed"
- The procedural cubemap was a real (~80 lines of C#) effort vs asking *"can you drop a CC0 HDRI in `Assets/HoloBorn/Lighting/`?"* — which would have been the right opening move

**Rule for next time:**

When approaching a fix that has both a "Claude-only path" and a "Claude+human path," **default to evaluating both before committing to the Claude-only path.** If the human action is <2 minutes and the result is structurally better (like a real GGX-prefiltered HDRI vs a procedural box-filter approximation), ask for the human action first.

The framing flip: not *"what can I build alone?"* but *"what's the right fix, and what's the minimum human collaboration to get there?"*

This isn't a hard mistake — the procedural cubemap turned out to be a useful fallback path baked into IBLBootstrap and validated the overall RenderSettings wiring before we had the HDRI. But ~90 min of iteration on Round 1/2/3 was sunk into tuning values that would be irrelevant once the real HDRI arrived. Time-budgeted poorly.

## 3. Sandboxed Desktop access kept biting (ecosystem observation, not a mistake)

**What happened:**
Across the session, Parthiv took 3-4 screenshots that I couldn't read because macOS TCC blocks Claude Code's Bash process from accessing `~/Desktop/` without Full Disk Access permission. The screenshots either landed in `/var/folders/.../NSIRD_screencaptureui_*` (clipboard route, sandbox-blocked) or `~/Desktop/` (TCC-blocked).

We worked around it by Parthiv dropping the critical Meshy WebUI screenshot directly into the repo root, where I could Read it.

**Why this is worth saving:**

Future Claude sessions on this machine will hit the same wall. The user can either:
1. **Permanent fix** — System Settings → Privacy & Security → Files and Folders → grant Terminal access to Desktop (or Full Disk Access)
2. **Per-screenshot workaround** — Cmd+Shift+4 (no Ctrl) to save to `~/Desktop/` AND grant Terminal Desktop access, OR drop screenshots directly into the repo (anywhere under `/Users/digispoc06/Documents/holoborn-server/`)

**Rule for next time:**

If a user mentions a screenshot is on `~/Desktop/`, **don't assume your Read tool can access it.** Verify with a quick `ls -la ~/Desktop/` first. If it returns "Operation not permitted," ask the user to drop the file into a project subfolder instead of fighting the sandbox.

Add this to the standing pre-flight whenever a session involves screenshot exchange with the user.

## Throughline across today's mistakes

Both #1 and #2 share the same anti-pattern: **acting on partial information instead of asking one clarifying question.**

- Mistake #1: didn't ask "which build is this?" before diagnosing
- Mistake #2: didn't ask "can you download an HDRI?" before building procedural

The clarifying question costs <30 seconds. Each missed question cost 30-90 minutes of iteration that would have been irrelevant if the question had been asked first.

**Rule that addresses both:**

> Before committing to a multi-step fix path, ask: *"is there a 1-question check that could redirect this entirely?"* Most useful checks: (a) what code is in the current build / what version did you test against, (b) what assets / tools / external state is already available that I'm assuming away.

This is yesterday's `feedback_test_matrix_before_proposing.md` applied at a higher altitude: not just track state-to-outcome mappings, but **verify state assumptions before generating fixes for them.**
