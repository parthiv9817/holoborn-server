# 2026-05-30 (Saturday) — Mistakes & lessons

Demo-polish + live-e2e day. Five lessons, three of them cost real iterations (and one bit *during* a live founder demo).

---

## 1. Unity: serialized scene values silently override C# field defaults

**What went wrong:** Changed the headline color (cyan → white), font size, and glow in `StageHeadlineController.cs`'s public field defaults. Built, sideloaded — and the headline rendered **still cyan with the same blocky glow.** It looked like the edit didn't take. (The founder was looking at the *old* cyan the whole time.)

**Root cause:** `StageHeadlineController` is a component **placed in the scene**, so its public fields are **serialized in `SampleScene.unity`** (Inspector values). At runtime Unity uses the **serialized scene values, not the C# defaults.** My code edits were dead on arrival. (Tell: the mandala *ring* color *did* change — because that component is added at runtime via `AddComponent`, so it used the code default. The headline didn't, because it's a scene object.)

**Rule:** For any serialized `MonoBehaviour` field on a scene/prefab object, **change the value in the scene/prefab, not just the script default.** Before assuming a value edit "didn't work," grep the `.unity`/`.prefab` for that field name — if it's serialized there, that's the value that wins. (Found them at `SampleScene.unity:2167-2177` and fixed there.)

## 2. Unity: "script class layout is incompatible between editor and player" after external edits

**What went wrong:** Edited scripts from outside the Editor (via file tools), then built → **"Error building player because script class layout is incompatible between the editor and the player"**, 6 errors, build failed in 4s. No actual `error CS` in the log — so not a compile error.

**Root cause:** The Editor was running a **stale in-memory script assembly** (pre-edit layout) while the player build produced the new layout. A `Cmd+R` / Asset Refresh **did not** re-sync them. Only a **full Unity restart** (clean recompile + domain reload) fixed it.

**Rule:** After editing scripts **externally**, force a full recompile before building — and if the layout-incompatible error appears, **restart Unity** (a refresh isn't enough). Read `~/Library/Logs/Unity/Editor.log` to confirm it's the layout error vs a real `error CS` before reaching for the restart.

## 3. OpenAI billing hard limit hit *during* the live founder demo

**What went wrong:** Run 1 of the live e2e (founder wearing the headset) failed at the view-synthesis step: `400 billing_hard_limit_reached`. The first OpenAI call (the portrait) succeeded and consumed the last of the budget; the **second** call (the 4-view grid) tipped the account over its hard limit. Demo stalled in front of Vipin.

**Root cause:** No pre-flight on OpenAI billing headroom before a known-cost live run. The Hunyuan path makes **two** OpenAI image calls (portrait + view grid) — the budget has to cover both. Same class of failure as the 2026-05-09 billing block; it recurred.

**Rule:** **Pre-flight the OpenAI billing headroom before any live e2e** — a cheap 1-token probe call confirms the account isn't at its hard limit (account-wide, so it predicts the image calls). Budget for *all* the OpenAI calls in the path, not just the first. For a founder/live demo, verify spend headroom in the same breath as warming the GPU.

## 4. Blind UI placement → an avoidable extra iteration

**What went wrong:** Placed the progress-% element while adb was disconnected (couldn't see the live frame). First guess (`y=-0.52`) put it far below the ring, mannequin-occluded; corrected to `+0.28`, which sat on the top rim ("too up"); finally `+0.16` once a real capture was pulled.

**Root cause:** Committing a pixel-placement decision without the visual, because the headset wasn't reachable. Each guess = a build + sideload cycle.

**Rule:** For UI **placement**, get the actual on-device frame before committing the value. Directional fixes (bigger, recolor) are safe blind; exact placement is not — a captured frame turns 2-3 guesses into one.

## 5. Know which git branch the running server is actually on

**What went wrong:** Briefly stated "decimation doesn't exist" — when it *does* exist, on the `feat/decimation-pre-rigging` branch. The live server was on `main` (no decimation), so the e2e ran the full ~471k-tri path. Both statements were partially true and it caused confusion mid-demo about whether decimation was active.

**Root cause:** Conflating "the code exists in the repo" (true — on a branch) with "the running server executes it" (false — running `main`).

**Rule:** When reasoning about live behavior, check the **checked-out branch of the running process**, not just whether code exists somewhere in git. State it precisely: "on `main` (live), not on the feature branch." `git branch --show-current` + `ls` the working tree before claiming presence/absence.

### Meta
Most of today's drag was Unity-editor/build-state friction (serialization, stale assembly) + the recurring OpenAI billing gap — all **pre-flightable**. The pipeline logic itself was clean end-to-end. As the project moves to maintenance, these are the operational checklists worth keeping: *change serialized values in the scene; restart Unity after external edits; probe OpenAI billing before live runs; capture-before-placing UI; verify the running branch.*
