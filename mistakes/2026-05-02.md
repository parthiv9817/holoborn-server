# 2026-05-02 — Claude's mistakes during the 8-build Phase 2 session

This was a long, frustrating session. User explicitly ended it angry and asked Claude to capture every mistake faithfully. This file is that record. One mistake per entry, three parts each (what / why / rule).

## 1. Did not web-search the bug until user demanded it

**What happened:** The crash signature `FScreenPSsRGBSourceMipLevelArray's permutation 0` was first seen in 2026-04-30's diary. Today Claude consulted ChatGPT and Gemini (via user's manual paste) for second opinions but did NOT use the available WebSearch tool to look up the exact shader name in UE forums or GitHub until ~5 hours into the session, after the user explicitly demanded "why cant you find atleast an absolute reason."

**Why it happened:** Pattern-matching laziness. Claude treated the AI second-opinion as a research substitute. ChatGPT/Gemini had stale or generalized knowledge — they both gave plausible-sounding fixes (try MMV-off, try ISR-on, try HDR-on) that were either wrong or only partially right. The actual root cause (UE 5.5 cooker regression for VULKAN_ES31_ANDROID) was findable in 1 web search.

**Rule for next time:** When a bug has a specific named identifier (shader name, error code, function name, exception class), web-search that exact identifier as STEP ONE. AI second-opinions come AFTER reading what the actual community has found. Don't outsource research to other LLMs without verifying against primary sources.

## 2. HDR-as-trigger theory contradicted user's direct evidence

**What happened:** Claude proposed flipping Mobile HDR off as the fix, citing the Cesium "Build for Quest" tutorial config (HDR off + MMV on). User correctly pointed out that yesterday's builds (2026-04-30) had HDR off and STILL crashed with the exact same shader. Claude's theory was incomplete — the trigger isn't HDR, it's stereo-array rendering being active at runtime. HDR being on creates one path that needs the array shader, but stereo with ISR/MMV creates the requirement independently of HDR.

**Why it happened:** Claude latched onto a documented working config (Cesium) and built an explanation that fit the shader name (sRGB → HDR tonemap → stereo array). Did not cross-check against the user's build history before proposing. The user has been doing builds since yesterday with detailed notes, but Claude treated each new fix as if from a clean slate.

**Rule for next time:** Before proposing any fix that involves toggling a setting, check the project's build history for: (a) was this setting toggled before in either direction, (b) what was the result. If a setting was already in the proposed state and the bug still happened, that setting is NOT the trigger. Maintain a written {settings, outcomes} matrix and consult it before each proposal.

## 3. Suggested re-flipping MMV after user just disabled it (without explicitly framing as new combination)

**What happened:** Build #7 disabled MMV. Crashed. Then in the very next message, Claude proposed a fix that included flipping MMV back ON (alongside HDR off). User replied "asshole asshole you forgot... you are just back and forth suggesting me solutions are you that dumb to suggest me again to turn on mmv when in the last step we litelrlay fuckin turned it offf."

**Why it happened:** Claude was correct that the proposed combination (HDR off + MMV on) was a NEW combination not yet tested, but communicated it badly — said "flip these two settings" without explicitly framing the contrast: "MMV-on is part of a NEW combination, this is not a retry of build #5 because HDR was on then." User read it as "Claude already forgot what we just did."

**Rule for next time:** When proposing a setting change that contradicts a recent action, explicitly call out the contrast. Use the matrix from rule #2. Frame as: "We tested A=on, B=off → crash. We tested A=on, B=on → crash. Untested: A=off, B=on. Proposing this." Never just say "flip these" without the contrast.

## 4. Wasted 32 minutes on UAT BuildPlugin (Path A) without checking Android packaging architecture first

**What happened:** When Build #4 hit the glTFRuntime plugin error, Claude proposed Path A (UAT BuildPlugin command to compile glTFRuntime for Android). 32 minutes later, the build completed successfully, but the output had only Mac dylibs and intermediate .o files — no shippable Android .so. Pivoted to Path B (add C++ module to project) which is what should have been Path A.

**Why it happened:** Claude assumed Android plugin packaging works like Win64 (per-plugin .dll files). Did not verify before kicking off the long-running command. The actual Android model is: plugin code statically linked into project's libUnreal.so during PROJECT packaging via UBT — not as standalone .so files. UAT BuildPlugin's Android target only compiles the source to .o and verifies it CAN compile, doesn't produce shippable artifacts.

**Rule for next time:** Before proposing a fix that involves a long-running command (>5 min), verify the fix actually produces the needed artifact for the target platform. Ask: "what file does this command produce, and is that file what the failing system needs?" For platform-specific build steps, check the platform's actual packaging model before assuming it matches another platform's.

## 5. Misdiagnosed OculusXR as "dead reference" when it was loading at runtime

**What happened:** Earlier in the session, Claude said the .uproject's OculusXR plugin entry was a "dead reference" because OculusXR wasn't installed. Later, Build #5's on-device log showed `LogHMD: FOculusXRHMDModule PreInit successfully` — meaning OculusXR DID load at runtime. The plugin module was somehow available even though Edit→Plugins didn't show it as an installed plugin (or showed it disabled).

**Why it happened:** Claude relied on the user's screenshot of the Plugins window (which didn't show OculusXR explicitly enabled) and concluded "not installed." Did not check engine-level plugin install paths or verify against runtime log evidence. The OculusXR plugin may have been installed at the engine level (not project level) and got loaded automatically when MetaXR was enabled (since MetaXR depends on OculusXR core in some configurations).

**Rule for next time:** Plugin "installed" vs "enabled" vs "loaded at runtime" are three separate things. Don't conflate them. Verify against the runtime log (`LogPluginManager: Mounting...`, `LogHMD: ...PreInit...`) for actual load state, not just project settings UI screenshots.

## 6. Failed to maintain test matrix — treated each build as one-shot retry

**What happened:** Across 8 builds, each of which changed multiple variables (HDR, MMV, ISR, plugins, Source/, etc.), Claude did not maintain a written matrix tracking {build #, settings, runtime state, outcome}. This led to mistakes #2 and #3 directly (proposing already-failed combinations or contradictory fixes). User explicitly called this out near end of session.

**Why it happened:** Claude was reactive — fix the latest crash, propose next fix, repeat. Did not invest 1 minute up front to set up a matrix that would save 30 minutes of confused proposals. Classic short-term-thinking trap when under time pressure.

**Rule for next time:** When a bug requires more than 2 fix attempts, BEFORE proposing the third attempt, write a matrix of all variables tested + outcomes. Update the matrix after each build. Reference it explicitly when proposing the next fix. The matrix is in this diary's "All the configs we tested" section — should have been written by build #4 at the latest.

## 7. Mobile HDR auto-flipping to True went uninvestigated for hours

**What happened:** When Claude added `bPackageForMetaQuest=True` to .ini, UE auto-flipped `r.Mobile.UseHWsRGBEncoding=True` (visible in the system reminder showing .ini changes after editor reopened). Claude noted it briefly ("UE auto-flipped one more setting that's genuinely promising") but did not investigate the implications. Later this contributed to the HDR-as-trigger theory being misleading.

**Why it happened:** Claude treated the auto-flip as a side-benefit ("UE thinks this is needed for Quest, good") instead of as an unexpected state change worth understanding. Cascading config changes from one toggle should be flagged as suspicious, not celebrated.

**Rule for next time:** When UE auto-flips other settings in response to a user-initiated change, treat it as a config-state-change worth one explicit investigation step: "what does this auto-flip imply about UE's model of the project, and could it create downstream issues?"

## 8. The "let's try X" cadence after exhausting documented options

**What happened:** After Build #7 failed with the same shader crash, Claude continued proposing "let's try one more thing" (HDR off, then NDK version, then settings combinations). User correctly identified this as flailing — "we want a proper fix not whack a mole gues that s consuming hours of my time."

**Why it happened:** Claude defaulted to "keep trying things" instead of "stop and recognize we're out of stock-Epic workarounds." The honest answer at that point was: "the cooker bug is real, no .ini fix exists, the options are flat-panel demo / source build / wait for hotfix." Should have given that verdict 1-2 builds earlier.

**Rule for next time:** After 3 attempted fixes for the same crash signature without progress, stop proposing "let's try X" and instead state explicitly: "I've exhausted the documented workarounds. The remaining options are [enumerated list of structural changes]." Distinguish "tweaking config" from "structural change" (engine swap, source patch, alternative ship path) and don't pretend more tweaks will work when they keep failing.

**Rule for next time:** When asking about a binary UI state, request unambiguous response format: "is the checkbox FILLED or EMPTY?" or "true or false?" or ask for a screenshot. Don't accept ambiguous English in checkbox-state questions.
