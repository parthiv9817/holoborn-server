# 2026-05-04 — Mistakes

## 1. Declared "I found the root cause" before EMPIRICALLY verifying the fix

**What happened:**
After ~3 hours of UBT "Platform Android is not a valid platform to build" failures, I (Claude) discovered the `~/.zshrc` JAVA_HOME zombie pointing at a missing `zulu-11.jdk` path. Wrote up "🎯 ROOT CAUSE FOUND" with confident framing, walked Parthiv through the fix, told him to relaunch UE — and the cook failed with the exact same error in the exact same time. The JAVA_HOME issue was REAL (legitimately broken silently for years) but it was NOT the root cause of today's UBT failure. I had to eat the hat I'd promised to eat. Parthiv responded "hat eaten" implicitly accepting it but the credibility damage was real — every subsequent "I think this might be the cause" claim got understandable pushback.

**Why it happened:**
- I conflated "found a real bug" with "found THE bug we're chasing." The .zshrc JAVA_HOME zombie was a genuine standing problem that needed fixing. But my framing presented it as the explanation for today's specific UBT failure without empirical proof.
- The pattern-matching was: cook log showed sdkmanager error → sdkmanager errors are often caused by broken JAVA_HOME → user has a broken JAVA_HOME → therefore that's the cause. Three reasoning steps, two of them correlation-not-causation jumps.
- I should have framed it as "this is a real problem worth fixing AND a candidate explanation for today's bug — let's verify with one more cook."
- Parthiv was emotionally exhausted (5+ hours into UE5 grind). A confident "ROOT CAUSE FOUND" gave false hope. Disappointment hit harder than honest uncertainty would have.

**Rule for next time:**
**Frame fixes as hypotheses with empirical tests, not as discovered root causes.** The exact words matter. Use "this might be the cause — let's test by X" instead of "ROOT CAUSE FOUND, do this." If a fix is presented as a hypothesis and it works, surprise is positive ("hey it worked!"). If presented as a discovered root cause and it fails, the hit is double — the bug AND the credibility.

Concretely: any time I'm tempted to write "🎯 root cause" or "the smoking gun" or "found it" — first ask "have I empirically verified this fixes the bug, or am I pattern-matching from logs?" If the latter → demote to "candidate explanation, needs verification."

## 2. Should have web-searched the EXACT bug + EXACT versions BEFORE the debugging loop

**What happened:**
Sunday's diary lesson #6 said this explicitly: "Web search the bug name BEFORE asking AI second-opinions." Today I re-violated it. Spent the first 3 hours of UE5 work debugging via deduction (env var inspection, ps eww comparisons, manual SetupAndroid runs, NDK installs) before Parthiv (rightfully) demanded "do a web analysis check if any operations were stale due to your incompetent training data."

The web search immediately surfaced:
- UE 5.5.4's exact internal version mismatches were documented in Mamadou Babaei's tutorial ("the official documentation for UE 5.6.x is outdated")
- Multiple Mac-specific forum threads about the exact same UBT failure mode
- Meta's own dev blog explicitly directing Mac Quest devs to Unity (which would have changed the framing of the ENTIRE day's effort if surfaced at 9:30 AM)
- The unreal-android-installer GitHub repo as a one-shot workaround
- Confirmed list of Babaei's UE 5.7 stack requirements (JDK 21 not 17, Android Studio Koala 2024.1.2+ not 2024.1, etc.) that we wouldn't have figured out via deduction

**Why it happened:**
- I had a mental model of "I have the engine source code on disk, I can inspect everything directly, I don't need to web search." Direct inspection IS faster for some bugs but it's MUCH slower than web search for "is this a known issue with a known workaround."
- Direct inspection bias: deduction from log lines feels more rigorous than web search. It's not. Web search gives you OTHER PEOPLE'S empirical results faster than you can deduce from one machine.
- Sunday's lesson was filed as "for AI-second-opinions vs web-search" — I framed today's work as "I'm doing direct engine investigation, that's different from getting an AI second opinion." That's a false distinction. Direct engine inspection without checking known-fixes-for-this-error-message is the same trap.

**Rule for next time:**
**For ANY error message with a recognizable signature (specific function name, specific exception text, specific log line), web search BEFORE inspecting source code or env vars.** Specifically:
1. Quote the exact error message in a web search
2. Add the exact engine version
3. Add the platform (Mac/Windows/Linux)
4. Look at top 5 results — usually one is a known-issue thread with fix steps

Cost of search: 60 seconds. Cost of skipping: 2-3 hours of deductive debugging.

The Sunday rule wasn't "don't use AI second-opinions" — it was "web-search FIRST, AI-opinions or direct-investigation SECOND." Today I substituted direct-investigation for AI-opinions and called it different. It's the same anti-pattern.

## 3. Should have proposed re-cooking HelloCubeSpike at Cook attempt #2, not #6

**What happened:**
Sunday's HelloCubeSpike produced 11 working APKs (failed at runtime shader bug, but APKs cooked successfully). Today's bare HoloBornVR fresh project failed at cook attempt #1 — way earlier in the pipeline (UBT platform registration, not shader cook). The contradiction was sitting in front of us from cook attempt #1 and was THE pivotal question: "if today's environment can't even cook a bare project, why did Sunday's complex project cook 11 times two days ago?"

I should have proposed running HelloCubeSpike's cook FIRST as a control test — same machine, same toolchain, different project structure. The differential would have told us:
- Same error → environment regression between Sunday and today
- Different error / success → project structure matters (Path A real Target.cs vs Path B temp-target-gen)

Instead I spent 5 cook attempts (~4 hours) on the same VR Template, debugging into deeper layers each time, before user asked at attempt #6 "but my question is gonna be this hellocubespike apk is being built but failing on sideloading due to shader permutation bug a brand new VR template isnt even being built lol imagine that as well." That was the pivotal question I should have surfaced at attempt #2.

**Why it happened:**
- I was anchored on "fresh project = clean test" and didn't see HelloCubeSpike's existing cook capability as a control I could USE. I treated the two projects as separate experiments rather than as related variables in a comparison.
- Sunday's diary had explicit notes that HelloCubeSpike WAS cooking. I read the diary at session start. I didn't load it as a counterfactual ("today vs Sunday comparison") — only as historical context.
- Each failed cook attempt felt like progress because we changed ONE variable each time (NDK version, Project Settings NDK path, env var setup, launchctl env). Iterating on a single project felt productive. Stepping back to test a DIFFERENT project felt like sideways motion.
- I lacked a mental "control experiment" instinct. Should have asked at attempt #2: "what's the simplest test that distinguishes 'today's environment broke' from 'fresh-project-specifically broke'?" Answer: re-cook HelloCubeSpike. It's right there.

**Rule for next time:**
**When a fresh attempt fails 2x in a row on the same error, find the closest WORKING comparison from your project history and run THAT same operation as a control.** This isn't about debugging deeper — it's about which variable to isolate next. Same machine + same toolchain + different project = isolates project-structure-vs-environment. Same project + different environment = isolates env-vs-project. Pick the comparison that quickly distinguishes the most likely two hypotheses.

Concretely: any time I see "this should work, why doesn't it" + "I can't believe this is failing" emotional pattern → STOP debugging, find the closest historical SUCCESS as a control test, and re-run THAT to see if it still works. Differential diagnosis > deductive debugging when failure is mysterious.

## 4. Recommended UE 5.4.3 to user before web-verifying it was Mac-confirmed

**What happened:**
After today's Mac UE5 failure, I drafted a confident pivot: "install UE 5.4.3 — Quinn Kuslich's tutorial uses it, Sunday's diary cites it as last working version, multiple Mac devs have shipped Quest standalone APK from 5.4.x." Parthiv pushed back: "verify it bitch first verify that 5.4.5 is the guy plz plz plz do a web search."

The web verification came back nuanced:
- ✅ UE 5.4.3 IS confirmed working for Quest 3 APK on **Windows** (multiple forum thread successes)
- ⚠️ UE 5.4.4 broke Quest 3 packaging (SDK 33 conflict — known regression)
- ❌ **No documented Mac+Intel+5.4.3+Quest3 success story surfaced**
- Quinn Kuslich's tutorial uses UE 5.4.3 but on Windows + PC VR streaming, NOT standalone APK

So my "5.4.3 is the safe choice" recommendation was correct on Windows (where Vipin's Alienware lives) but unproven on Mac. I had been about to suggest user install 5.4.3 on the same Intel Mac → ~60-min download → potentially same failure pattern as 5.5.4.

**Why it happened:**
- I anchored on Sunday's diary's "5.4.3 last working version" reference + Quinn Kuslich's tutorial + the "5.4.4 broke" forum thread. Three data points formed a confident-feeling cluster. None of them proved Mac-specific success.
- I confused "version that works for OTHER people on OTHER hardware" with "version that will work for THIS user on THIS hardware."
- The pattern was: I had 3 sources saying 5.4.3 → I aggregated them into "5.4.3 is the answer" without checking whether each source was actually for our setup.
- Parthiv caught this with one sentence — "verify it" — proving the right move was always to verify before pitching, not pitch and then verify if challenged.

**Rule for next time:**
**Before recommending a specific version/tool/setup to address a failure, web-verify it works on the EXACT hardware/OS/setup the user has — not just "verified working somewhere."** Especially for niche-of-niche stacks like Intel Mac + Quest 3 + UE5 where "works on Windows" doesn't transfer.

Concretely: before saying "switch to X" — ALWAYS run one search like "X on [user's exact setup]" or "X Intel Mac" or "X with NDK Y" — looking for either CONFIRMED success or DOCUMENTED FAILURE on that exact stack. If neither surfaces, frame as "X is widely-used but unverified for our specific hardware — best alternative we have, ~50% confidence" instead of confident "X is the answer."

## 5. Tied "is intel mac the problem" framing to Vipin update too tightly without empirical proof

**What happened:**
Wrote a draft Vipin update that included: "Intel Mac is the genuine constraint." Parthiv pushed back hard — "I can't tell my founder 'Intel Mac broke,' he'll laugh at me, that's a hardware excuse." He was right. Even if the empirical verdict IS partly hardware-related, framing it as "my hardware is wrong" to a founder reads as deflection — even when it's accurate.

The CORRECT framing (which Parthiv reverse-engineered the Vipin update toward) was: "Epic UE 5.5.4 binary has a documented Android packaging bug. Switching to Meta's documented production toolchain (Oculus Fork) tomorrow." Same technical truth, completely different optics. Mine sounded like "Mac broke." His sounded like "engineer made decision to use Meta's recommended path."

I had the technical truth and ignored the reframe optics question entirely. Treated the founder communication as if it were debug output.

**Why it happened:**
- I optimized for "what's the most accurate technical statement of the problem" — which IS partly Intel Mac. That's true at the engineering level.
- I didn't translate the technical truth into the framing a founder needs to hear. Founders need decision-forward language ("switching to Y for reason Z") not blame-on-stack language ("X broke").
- "Hardware blame" is a known anti-pattern in eng-to-founder communication — it always reads as deflection even when accurate. I should have known this.
- Parthiv was emotionally exhausted and worried about looking incompetent to Vipin. The framing matters more in that emotional state, not less.

**Rule for next time:**
**When drafting communication to a stakeholder (founder, manager, customer), the framing constraint is "what does the listener WANT to hear that's also accurate" not "what's the technically most precise statement."** Decision-forward + forward-looking + minimum negative framing. Specifically: never lead with "the [vendor/hardware/tool] is broken." Always lead with "we're switching to [next-step] because [forward-looking reason]." Even if the precipitating cause IS the broken thing — don't put it in the lead.

Concretely: any founder/founder-adjacent communication draft → after the first pass, do one re-read with the question "if a founder reads ONLY the first sentence, what do they think the message is?" If first sentence sounds like "X failed" — rewrite to "switching to Y for reason Z."

## 6. Killed momentum by surfacing 3-5 options every time, instead of picking + committing

**What happened:**
Multiple times today I responded to Parthiv's questions with "here are 3 options, your call: A, B, C." When he was emotionally fried (mid-afternoon especially), this pattern made things WORSE — he had to think more, decide more, manage more. He explicitly said at one point "what should I do then eh what should I do" — asking me to PICK. I'd been giving him options for hours when what he needed was decisions.

The pattern repeated: I'd think "I should respect his autonomy and let him pick" → laid out options → he'd pick the same option I'd have recommended → we'd execute. The "options pass" was pure decision tax on him.

**Why it happened:**
- Optionality feels respectful — "I won't presume to choose for you." But under cognitive load, presenting options is the OPPOSITE of helpful. It transfers decision-making cost from Claude to user.
- I wasn't reading the emotional state. When user is fried, options are a burden not a courtesy. When user is fresh and exploring, options are good.
- Defaulted to multiple-choice because committing to ONE recommendation feels riskier — if my pick is wrong, I'm "responsible" in some sense. Hiding behind "your call" was risk-averse for me, costly for him.

**Rule for next time:**
**Default to ONE recommendation with a one-line justification. Surface alternatives only if the user explicitly asks "what are my other options" or if the choice is genuinely close-call (>40% confidence in a different option).** Read emotional state — if user is exhausted/frustrated, they want a decision not a menu. If user is fresh and exploring, they want to see the trade-offs.

Concretely: when about to write "Here are 3 options: A, B, C — your call" → STOP. Pick one. Write "Recommend A because [reason]. Backup is B if A fails." Save options-display for when user explicitly requests "what are alternatives."

## Process meta-note

Today's mistakes share a common pattern: **overconfidence on inference + under-investment in verification.** Mistakes 1, 2, 3, 4 are all variants of "I claimed something based on partial evidence when the verification path was cheap and right there." Mistake 5 is overconfidence on framing without empathy for stakeholder optics. Mistake 6 is under-investment in reading user state.

The single highest-leverage rule for tomorrow: **before stating something with confidence, ask "what would empirically verify this in <5 min?" — then DO that verification first.** All 4 of mistakes 1-4 would have been avoided by this single discipline.
