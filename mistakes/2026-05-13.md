## 1. Silent `LogWarning` fallback swallowed the structural bug for 6 hours

**What happened:**
`TestGlbLoader.LoadGlbAtAsync` had a private `isLoading` field that rejected concurrent calls. When Brick 3's retex preload (Stage 1 entry) was still in flight at the moment Brick 4's rigged preload tried to fire (Stage 2 entry), the second call was rejected with `Debug.LogWarning("already loading; refusing concurrent call")`. `Stage3ReleaseController` saw a null Task and fell into its fallback path: animating the static retex avatar as if it were the rigged GLB.

This fallback was logged via `LogWarning`, NOT `LogError`. Quest sideload doesn't surface `LogWarning` in any visible UI by default — you only see it via `adb logcat`. So for 4 consecutive council audit rounds (rounds 1-4), the "rigged avatar" we were debugging visually was actually a static retex avatar with no skeleton, no Animation component, and no animation clip.

Every council finding from rounds 1-4 was technically correct but diagnosing symptoms on a structurally broken foundation:
- "Squat invisible" → retex has no skeleton, `FindBone` returned NULL for every bone
- "No idle anim" → retex has no Animation component
- "Weird stride pose" → didn't exist; we were seeing retex's static A-pose
- "Fall not visible" → retex was already at floor level from Stage 2 hold

**Why it happened:**
- I wrote the `isLoading` guard defensively for the legacy press-B path (which shares `currentAvatar` state), but added the same guard to `LoadGlbAtAsync` without thinking about whether concurrent async loads actually conflict (they don't — each call owns its own GltfImport + GameObject)
- The LogWarning was added "just in case" without considering that Quest sideload doesn't surface warnings
- The fallback path (`_riggedAvatar = retex`) was supposed to be a graceful degradation for "rigged preload failed in production" — but it also fired silently when "rigged preload was rejected by your own code"

**Rule for next time:**
**Silent fallback paths in integration code must be LOUD or REMOVED.** Specifically:
- If a code path falls into a degraded mode in production, log it as `LogError` (visible in any logcat filter), not `LogWarning`
- ALSO surface it in UI / status text where the operator can see it (Parthiv had no on-device way to know the fallback was firing)
- Better: REMOVE silent fallbacks entirely. Throw an exception or refuse to continue. If the rigged GLB is required, missing it should hard-fail visibly, not silently degrade.

This is the single biggest time-cost of the day. Probably 4-5 hours of debugging would have been avoided if the rejection had been LOG_ERROR + visible status text saying "Stage 3 disabled: rigged GLB preload failed."

Compound learning: **the prior project's `feedback_test_matrix_before_proposing.md` rule applies at the integration layer too.** Before iterating fixes for visible symptoms, verify the structural assumptions are intact (which GLB is actually loaded, which animation is actually playing, which bones actually resolved). One logcat capture at round 2 would have caught it 4 hours earlier.

## 2. Procedural biomech was probably the wrong abstraction from the start

**What happened:**
Brick 4's design called for an "impact squat" — knees folding, hips dropping, spine curling on landing. I implemented this as procedural bone rotation in C# (`Quaternion.AngleAxis` per-frame on Hips / UpLeg / Leg / Spine bones). Compound coroutine with compression + hold + recovery phases, configurable magnitudes, fuzzy bone matching.

Parthiv later recalled his previous Unity project: he'd drag a Mixamo "Falling Idle" → "Hard Landing" clip into Unity, drop it into the Animator, done. The clip's keyframes ARE the biomech, hand-animated by Mixamo artists. Zero procedural code, perfect result.

We never got to test the procedural-only path cleanly because of mistake #1 — but even if we had, ChatGPT's audit in round 6 noted that Meshy's auto-rigging often has non-standard local bone axes, making procedural rotation around `Vector3.right` unreliable. The procedural approach was fighting the rig's conventions.

**Why it happened:**
- Initial scoping: I treated "find the right Mixamo clip" as harder than "code up procedural biomech" because Meshy's animation library is opaque and `action_id` labels are unreliable
- I underestimated how much hand-animated knowledge is baked into a Mixamo landing clip (anticipation, inertia, pelvis lag, shoulder bob, weight transfer) — none of which we could practically recreate procedurally in 30 min
- I framed clip-based as "Option C: phase 2 polish" instead of "Option C: actually the right architecture from the start"

**Rule for next time:**
When a target domain has **pre-built hand-animated artifacts** (Mixamo clips, motion capture libraries, etc.), do NOT try to procedurally recreate them in code. Use the artifacts. Procedural physics is the right abstraction for emergent behavior (ragdolls, IK targets, physics responses) but NOT for replicating intentional human motion. The 80/20 of human-like landing is in the clip's keyframes, not in math.

Specifically: when Parthiv mentioned "in my old Unity build I had Mixamo avatars that fell with proper joint folding," that was the primary architectural signal. I framed it as "interesting fallback" instead of "you've just told me the correct approach."

## 3. Motion-amplitude metric optimized for the wrong dimension

**What happened:**
Backend Zoro ran a quantitative motion-amplitude metric in parallel against 3 candidate "idle" clips (action_id 12, 244, 38). Dozing_Elderly (38) won the metric by a factor of 7-8× — lowest hip translation, lowest arm rotation, lowest feet rotation, lowest spine rotation across the board. We staged it as `test_rigged.glb`.

Sideload + visual inspection: Dozing_Elderly is unequivocally a **chair-sitting pose**. The metric was technically correct — it WAS the lowest-motion clip in the candidate set — but "lowest motion" naturally selected the most STATIONARY pose, which happened to be a sleeping/seated mocap. We optimized for "least motion delta" but needed "standing pose + low motion."

**Why it happened:**
- The metric was a clever response to Meshy's unreliable labels (which had already burned us once with `action_id=0` and `action_id=11` both being walking cycles)
- It correctly modeled "I want a clip that doesn't move around much"
- It failed to model "AND the rest pose itself should be standing upright"
- Backend Zoro flagged the uncertainty ("absolute magnitudes too large to be meters of body motion, possibly scene-level coordinates"), but the relative ranking was clear
- Neither of us thought to ALSO check the bind pose / frame 0 of each clip visually before staging

**Rule for next time:**
**Motion amplitude ≠ posture class.** When selecting motion data for "idle/standing" use cases, the metric needs to model BOTH:
1. Low motion delta over time (what we measured) ✓
2. Standing posture at any frame (what we didn't measure)

Practical fix: when selecting from an unfamiliar animation library, ALWAYS visually preview frame 0 of each candidate before staging. Backend Zoro had downloaded the preview GIFs earlier — should have opened them in Finder before picking. The metric narrows the candidate set; visual confirms the final pick.

Compound learning: when designing a selection metric, ask "what could optimize for this metric but be wrong?" before committing. In this case: "what kind of clip has very low motion BUT isn't useful for our purposes?" Answer: a sleeping clip. Should have been obvious in retrospect.

## 5. Over-iterated council rounds without questioning the foundation

**What happened:**
Six council audit rounds today across Brick 3 (2 rounds) and Brick 4 (4 rounds before logcat-fix). Each Brick 4 round produced detailed findings — bone names, animation override theories, axis hypotheses, easing curve recommendations. None of them were wrong in isolation, but ALL of them were diagnosing symptoms on a structurally broken pipeline (mistake #1).

Round 4 council gave a **direct contradiction**: Gemini said the impact squat was firing weakly; ChatGPT said it was completely rigid (not firing). When two competent AI auditors give opposite diagnoses on the same video, that's an information signal: **the bug is below both of their hypotheses.** Should have escalated to logcat capture immediately at that point. Instead, I shipped a defensive bundle addressing both hypotheses (fuzzy bone match + autoplay disable + curve change + staging) and ran another round. Logcat finally got captured in round 5 and revealed the truth.

**Why it happened:**
- Council rounds had been productive across Brick 3 (rounds 1-2 caught real issues, fixed them, validated). I trusted the council loop to keep working in Brick 4.
- I treated each round's findings as "additional information" rather than "another hypothesis at the same layer of the stack"
- The defensive-bundle pattern (ship fixes for all stated hypotheses) felt rigorous but actually masked the underlying bug — each round's symptoms were partially mitigated but never eliminated
- I didn't have a "this council round contradicts the last" trigger for switching diagnostic mode (frame analysis → logcat)

**Rule for next time:**
**Contradictions between council AIs are a diagnostic alert, not a tie-breaker problem.** When two AIs give opposite reads of the same evidence, escalate the layer of investigation — don't try to satisfy both hypotheses with a defensive bundle. The bug is probably below both. Specifically:
- Frame analysis caught Brick 3's hidden bug (7s freeze)
- Logcat would have caught Brick 4's hidden bug (concurrent preload rejection) if I'd run it after round 2 instead of round 5
- After ~3 inconclusive iterations of "another fix bundle, another council round," switch diagnostic mode

This is a flavor of `feedback_test_matrix_before_proposing.md` but at a higher altitude: track HYPOTHESES and their outcomes too, not just settings/outcomes. If three hypotheses in a row land partial-fix, the layer is wrong.

**Why it happened:**
- Meshy's documentation links and naming conventions suggested honest semantics
- I treated their action_id catalog as a reliable index when it's clearly a noisy mapping
- Each "this one's better than the last" iteration felt like progress but was actually just a worse-or-similarly-wrong result

**Rule for next time:**
**External system labels in integration code are NOT trustworthy until visually verified.** Specifically for Meshy / similar animation libraries:
- Preview the clip visually BEFORE baking it into a graft pipeline that costs credits + minutes
- Backend Zoro had downloaded preview GIFs as part of the research-agent flow — should have been a mandatory step before any `action_id` re-bake, not an optional "if you want to eyeball"
- If a label says "Idle" but the metric or preview shows locomotion patterns, the label is wrong and the search must continue

Compound learning: `feedback_websearch_before_ai_opinions.md` (verify exact bug identifiers via web search before AI second-opinions) applies here as "verify external system labels via visual preview before committing iterations."

## 8. Should have added structured error/state logging from day one on integration points

**What happened:**
The integration layer between Unity (UI Zoro) and Backend (Backend Zoro) had multiple silent state transitions:
- TestGlbLoader's isLoading rejection
- Stage3 fallback to retex
- Animation's playAutomatically default
- glTFast bone-name handling
- Meshy's action_id semantics

Each of these had emergent behavior that only became visible via logcat (which we only captured at round 5). The intermediate cost was 4 council rounds of debugging on incomplete information.

**Why it happened:**
- I added logging in places that felt important to ME (controller events, state transitions) but not at the INTEGRATION POINTS where assumptions about external behavior could fail
- The bone-hierarchy auto-dump I added in round 3 was the RIGHT instinct — should have been a Day-1 reflex on any integration-heavy code, not a round-3 diagnostic add

**Rule for next time:**
**On any integration-heavy code path, ship structured logging at the integration points from Day 1.** Specifically:
- When loading external assets: log size, validity, bounds, attached components
- When applying procedural transformations to external rigs: log which bones resolved + which didn't
- When subscribing to external events: log first-fire timestamp and source
- When fallback paths fire: LogError not LogWarning, plus UI status text

Quest sideload + adb logcat is the diagnostic loop. Plan for that being the inspection surface, not the Unity Editor playmode (which masks plenty of integration-layer failures).

## Pattern across today's mistakes

**The throughline:** I trusted defaults / labels / silent fallback behaviors at integration points without verifying empirically.

Specific manifestations:
- isLoading guard's default behavior (silently reject) wasn't questioned
- Meshy's `action_id` labels assumed honest
- Mixamo bone naming convention (mixamorig: prefix) assumed standard
- Animation `playAutomatically` default assumed off
- Spine hierarchy assumed standard Mixamo (turns out reversed)
- "Idle" labeled clips assumed planted-feet

In each case, **empirical verification at the integration point would have saved hours.**

The compound learning from prior diaries: `feedback_test_matrix_before_proposing.md` (test matrix before proposing fixes) operates at the controller-state level. **Today extends it: also test ASSUMPTIONS at integration points before building on top of them.** Before assuming X about how Y system behaves, write a 5-line diagnostic that proves X is true, log the result loudly, ship that BEFORE building on the assumption.

**Rule that addresses all of these:**

> Before building on top of an external-system / framework-default behavior, ship a Day-1 diagnostic that proves the behavior matches your assumption. The diagnostic should log at `LogError` level (or surface in UI) if the assumption fails. Don't trust silent fallbacks. Don't trust unverified labels. Don't trust "the default does the thing I want."
