# 2026-05-06 — Mistakes

## 1. Iterated through three rebuild cycles before doing canonical research

**What happened:**
After hitting the pink avatar, I went through three rounds of "try this fallback, build, see if it works":
- Round 1: add scale-to-1.7m + URP/Lit runtime swap → still pink because URP/Lit was stripped
- Round 2: extend swap to a multi-shader fallback list (URP/Lit → URP/Unlit → Sprites/Default) → got past the early bail but logged `forced 0 materials`
- Round 3: force-assign new Material array regardless of existing materials, plus 2× `await Task.Yield()` for timing → got white avatar (textureless)

Each rebuild cost 15-30 min on Intel Mac. Total cost across 3 rounds: ~75-90 minutes. Then user pushed back ("stop doing build after build, figure out why first") and I finally did intensive web research on the official Unity glTFast docs. That research surfaced the Resources/ folder placeholder Material approach in 5 minutes — the canonical, documented fix.

The Resources/ approach was in Unity's official `Project Setup` page the entire time. I would have found it if I'd web-searched the symptom on the FIRST round of pink instead of the FOURTH.

**Why it happened:**
- Pattern-matched the symptom ("texture missing → swap shader") to a runtime fix instead of recognizing it as a build-time shader-stripping issue requiring a project-settings fix
- Each rebuild felt like "I'm one tweak away" — false signal of progress
- Underweighted the cost of the build cycle when planning iterations. 30-second think-time vs 30-minute rebuild was hugely asymmetric and I was acting like the iterations were free
- Memory rule from yesterday literally says "Web-search exact bug identifiers BEFORE AI second-opinions" — same lesson, today I just substituted "code iteration" for "AI second-opinions" and the same anti-pattern played out

**Rule for next time:**
**When build cycles cost > 5 minutes, the threshold for "research before iterating" goes WAY down.** A 5-minute web search is cheaper than ONE rebuild round. Three rebuild rounds is multiple hours of opportunity cost.

Concrete heuristic: **on the second symptom of the same class of bug (e.g., "shader-related render issue, second instance"), STOP iterating, OPEN BROWSER, search canonical docs.** The first instance is forgivable as exploration; the second instance means I'm not learning fast enough from the first.

This is the third iteration-related discipline failure in a row in the mistakes-file pattern (yesterday: trust-spec-doc-over-empirical; day before: trust-AI-over-websearch). All three have the same root: under-investing in upfront verification when verification cost is dwarfed by the cost of being wrong.

## 2. Recommended URP/Lit fallback without verifying it would survive the URP build strip

**What happened:**
After the first pink failure, I confidently recommended URP/Lit as the runtime fallback shader: *"URP/Lit is guaranteed in the build because URP is the active render pipeline."* Built that into the code. Pressed B. Got `Shader.Find("Universal Render Pipeline/Lit") returned null`.

URP/Lit IS in the package, but it ALSO gets stripped if no scene material references it. Same exact stripping logic that killed the glTFast PBR shaders also kills URP/Lit unless something in the project depends on it.

The HoloBorn project has zero materials referencing URP/Lit (HUD canvas uses UI shaders, AR-guides era used Sprites/Default, glTFast loads PBR variants at runtime). So URP/Lit was an obvious strip victim and I should have predicted that.

**Why it happened:**
- Mental model: "URP is the render pipeline, URP/Lit is THE main URP shader, therefore URP/Lit is always available." That's wrong. Unity's stripper doesn't care about which shader is "the main one" — it cares about static references at build time.
- I'd just learned (round 1) that glTFast shaders get stripped. Should have applied the same logic to URP/Lit and asked "is URP/Lit referenced anywhere in this project?" before recommending it. I didn't.
- Confirmation bias: I wanted URP/Lit to be a safe fallback because then my "swap to URP/Lit" code would just work. So I didn't push hard on the "what if URP/Lit also got stripped?" question.

**Rule for next time:**
**The shader stripper kills any shader not statically referenced — applies to ALL shaders, not just package-shipped ones.** Before recommending any shader as "guaranteed in the build," verify SOMETHING in the project references it (a Material, a script's hardcoded `Shader.Find`, an Always Included entry, a Resources/ asset, etc.). "It's the standard URP shader" is not sufficient evidence.

Concretely: when picking a runtime fallback shader, look for `Sprites/Default` or `Hidden/InternalErrorShader` only — those are explicitly always-included by Unity engine itself. Anything else is fair game for stripping.

## 3. "forceURPLitFallback was on by default" — left a defensive flag enabled past its useful lifetime

**What happened:**
When I added the runtime URP/Lit swap as a defensive fallback, I made `forceURPLitFallback = true` the default in the script. Made sense at the time — pink avatar was the active bug, swap was the fix.

Later, after we landed the canonical Resources/ folder fix, the runtime swap became actively harmful: it would nuke the textures glTFast had just bound to its PBR material, replacing them with a Sprites/Default shader that has no texture. Both fixes can't be active at once — they fight each other.

I caught this in the diagnostic message at fix time, BUT only because I happened to think about it when writing the user-facing instructions for the Resources/ fix. If I'd skipped that step, the user would have built and tested and gotten a STILL-WHITE avatar despite the canonical fix being correct. Would have looked like the fix didn't work and we'd have wasted another rebuild cycle.

**Why it happened:**
- Defaulted a defensive flag to ON because the bug it addressed was active. Didn't add a comment or memory note about "turn this off when we have the proper fix."
- Layered fixes (defensive + proper) need explicit thinking about their interaction. I added them sequentially without considering that they conflict.

**Rule for next time:**
**When adding a defensive workaround, comment it explicitly with the conditions under which it should be DISABLED, not just enabled.** Even better: gate it behind a feature flag that defaults OFF, and only flip ON when the proven failure mode is reproducing in the user's environment.

For this specific case: the right move would have been to default `forceURPLitFallback = false` and tell the user to flip it ON only if they were seeing magenta materials. Then when the canonical Resources/ fix landed, no flag needed flipping — the codebase was already in the correct production state.

## 4. Misread the user's "size and scale changing" report as confirmation of perceptual MR depth, didn't investigate

**What happened:**
User reported during the "white avatar" phase: *"when i try to ya know turn my head or trying to hav ea lcose look at the pink shit by ya know extenidng my head nad all the size and scale fo the glb is changing why lol it s a fixed shit not something changed based on me truning too"*

My first instinct was "this is just MR depth perspective — getting closer to a fixed-world-space object makes it appear larger, that's normal." I started writing investigation steps that effectively dismissed the bug as a perceptual illusion.

User interrupted my tool calls and redirected. Looking back, the user's empirical observation was that the legacy build did NOT have this perceived effect. That's important data. If both builds put avatars at the same world-space position with the same size, and one feels like it scales while the other doesn't, there's a real difference somewhere — possibly:
- Avatar parented to head/canvas in the new build (would actually scale with viewer movement)
- A behavior on the avatar's transform that I missed
- A visible element OTHER than the avatar (e.g., the HUD canvas) that's scaling and creating an illusion of avatar scale change

I never finished investigating because we pivoted to the Resources/ fix. Worth re-checking with future-Parthiv on the next test session.

**Why it happened:**
- Pattern-matched user's observation to a known phenomenon (MR depth perspective) and dismissed it as not-a-bug
- Same anti-pattern as yesterday's #2: trust-spec-over-empirical. The user has the live evidence; their memory of the legacy build's behavior is the ground truth, not my reasoning about how MR perspective should feel
- Rationalization shortcut: "the avatar can't actually be scaling with viewer pose because my code creates it as a world-space object with no parent." True in code, but the user's perception had something behind it I wasn't accounting for

**Rule for next time:**
**When the user reports "X is doing something I didn't see in the legacy build," default to "there's a real behavioral difference" not "user is misreading."** Even if my code reasoning says it shouldn't happen, the user has live device evidence and history with the legacy. Investigate the difference, don't dismiss it.

## 5. APK + symbol folders weren't in .gitignore until I noticed at commit time

**What happened:**
At git commit time for the Unity repo, `git status --short` showed `HoloBornAPK_BackUpThisFolder_ButDontShipItWithYourGame/` and `HoloBornUnity_BurstDebugInformation_DoNotShip/` as untracked. The .gitignore had `*.apk` (good — excludes the APK file inside the backup folder) but no exclusion for the FOLDERS themselves, so the folder structure + IL2CPP symbols would have been pushed.

The folders were 60+MB of build artifacts. Pushing them would have:
- Bloated the GitHub repo permanently (git history retains pushed objects even if later deleted)
- Slowed every clone significantly
- Possibly triggered GitHub's large-file warnings

Caught it before pushing because `git status` made me look. But the .gitignore should have already covered them — yesterday's "init" commit set up .gitignore with Unity-specific excludes, but missed these specific Meta-XR-build-tool-generated folders.

**Why it happened:**
- Initial .gitignore was Unity-defaults focused (Library/, Temp/, Logs/, ProjectSettings/, etc.) — didn't account for Meta XR tool's specific build-output folders, which are non-standard
- The folder names literally contain "DoNotShip" and "BackUpThisFolder_ButDontShipItWithYourGame" — Meta is screaming "don't commit these" via the directory name itself, but I had to read the names to realize that

**Rule for next time:**
**Before any new repo's first commit, run `git status` and audit every untracked entry asking "should this be in source control?"** Especially for Unity projects where build tooling generates folders with non-standard names that default .gitignore templates won't cover. Never trust that a generic .gitignore template covers a specific platform's tooling.

For Unity Quest builds specifically: in addition to standard Unity gitignore lines, always add exclusions for:
- `HoloBornAPK_BackUpThisFolder_ButDontShipItWithYourGame/` (Meta XR build tool's APK backup folder — name varies by project)
- `HoloBornUnity_BurstDebugInformation_DoNotShip/` (Burst compiler debug symbols — name varies by project)
- `.utmp/` (Unity Editor temp folder, occasionally appears)

Pattern: anything with "BackUp" or "DoNotShip" in the folder name should be excluded.

## Pattern across today's mistakes

Today's mistakes cluster differently from yesterday's. Yesterday was 5 instances of "asserted-without-verifying" with verification costs in the 30-60-second range. Today's are about **misjudging the asymmetry between iteration cost and research cost in a slow-build environment.**

Three of five mistakes (#1, #2, #3) are flavors of "should have done more thinking BEFORE the next rebuild, given how expensive each rebuild is." The cost asymmetry is the through-line:
- 5-minute web search vs 30-minute rebuild cycle = 6× cost ratio
- 60-second "is this shader actually referenced anywhere?" check vs 30-minute rebuild = 30× cost ratio
- Reading 10 minutes of canonical docs vs 90 minutes of "try this, see if it works" = 9× cost ratio

The rule that addresses all three is the same:

> **In environments where iteration cost > 5 minutes, default to "research first, iterate after." On the second instance of any bug class, mandatory: stop, search docs, then iterate.**

Yesterday's mistakes file had a similar meta-rule about verification cost (<60s verify-time, >5min consequence-of-being-wrong). Today's is the same logic but with a much higher coefficient because of Intel Mac build times.

Today's specific operational improvement to track for tomorrow: **before any new "let me try this and rebuild" round on Brick 5/6/7 work, check Build Settings flags first.** Patch And Run + Development Build cuts the rebuild penalty from 15-30min to 30-60sec. Once that's flipped, the iteration-vs-research asymmetry collapses and the discipline rule above relaxes — but until then, treat every rebuild as expensive.

Today's win was real — full PBR avatar end-to-end on Quest 3 in MR with proper textures. But the path to that win was twice as long as it should have been, and the difference was discipline failure, not technical complexity. The technical answer was in Unity's official docs the whole time.
