# 2026-05-09 — Mistakes

## 1. Assumed glTFast URP property names without verifying source code

**What happened:**
Built `BiasMaterialsForMatte` to fix the founder's "plastic figurine" complaint by multiplying the material's roughness factor (1.4x) and reducing metallic factor (0.5x). Used the property names `_RoughnessFactor` and `_MetallicFactor` — based on URP/Lit naming convention with a leading underscore.

Logged "BiasMaterialsForMatte: roughened 1 materials" — counted material iterations, not property hits. The HasProperty checks were ALL silently returning false, SetFloat was no-op'ing, and no actual bias was being applied to the glTFast Shader Graph variant.

User tested at 18:14 and 18:29 with multipliers (1.4/0.5 then 2.5/0.1). Marginal change at first → claimed "fix landed." Aggressive multipliers made the face go DARK instead of matte → claimed "over-correction, lighting issue."

It took 2-3 hours of iteration before the spawned research agent went to glTFast source code and found:

```csharp
// glTFast/Runtime/Scripts/Material/MaterialProperty.cs
public static readonly int Metallic        = Shader.PropertyToID("metallicFactor");
public static readonly int RoughnessFactor = Shader.PropertyToID("roughnessFactor");
// camelCase, NO leading underscore
```

Our code's `HasProperty("_RoughnessFactor")` was always returning false. The `_Smoothness` URP/Lit fallback branch was the only thing actually firing — and only inconsistently, depending on whether the glTFast variant happened to also expose URP/Lit properties.

**Why it happened:**
- I extrapolated from URP/Lit convention (`_Color`, `_Metallic`, `_Smoothness`) where leading-underscore + PascalCase is standard
- glTFast 6.x's URP Shader Graph uses glTF spec naming convention (camelCase, no underscore — matches glTF JSON keys directly)
- I never grepped glTFast's source code or read its API docs — assumed Unity convention applied
- Unity's `Material.HasProperty` and `SetFloat` both fail SILENTLY on unknown properties — no error log, no warning, no exception. They just return false / no-op.
- I trusted the "1 material touched" log as proof the bias was working — but that count was iteration count, not hit count. I didn't break it down per-property until the research agent forced me to.

**Why it dragged so long:**
- User reported "marginal change" between 1.4/0.5 and 2.5/0.1 — I attributed to "subtle multipliers, need more aggressive"
- When face went dark at 2.5/0.1, I attributed to "Quest MR has flat ambient, no probe, etc." (which IS true but wasn't the dominant cause)
- The visible difference between the two settings was the URP/Lit `_Smoothness` fallback inconsistently firing — which I didn't suspect because I trusted the camelCase path was working

**Rule for next time:**

For any third-party Unity package with custom shaders, **grep the source code for `Shader.PropertyToID("...")` calls before assuming property names.** Don't extrapolate from Unity's stock conventions.

For ANY `Material.SetFloat` / `SetColor` / `SetTexture` call, log AT LEAST one of:
- `mat.shader.name` — tells which shader variant Unity picked at runtime
- `mat.HasProperty(name)` result before the Set — proves the property exists
- A diagnostic dump of `mat.shader.GetPropertyName(i)` for all properties — full picture

I added all three in the round-3 fix. **Should have been there from the start of any material manipulation work.**

Pattern: this is the same shape as the Phase A "claim done before TestClient verify" mistake from 2026-05-08. **Verification at the API boundary** (HasProperty returning true) was needed before claiming downstream behavior (bias applied). Skipping that verification let me ship a silent-no-op for hours.

## 2. Predicted vortex visual outcome without device verification

**What happened:**
Designed `UpdateVortex()` with per-cube orbit phase computed via `VortexBehavior.ComputePhase(i, count, time, revsPerSec)` — gives each cube a unique angle around the orbit, evenly distributed.

Same code computed per-cube vertical drift via `(elapsed + verticalOffset) % cycleSeconds * driftSpeed` — where `verticalOffset = (float)i / count * cycleSeconds`.

Both formulas had the SAME linear correlation with cube index `i`. Result: cube i=0 was at orbit angle 0 with vertical 0; cube i=1 was at orbit angle 2π/N with vertical (1/N) × cycleSeconds; ...

This created a perfect HELIX. From the user's viewpoint in MR, the helix read as a "dragon-shaped chain of cubes" winding upward.

I had previously claimed in chat: "you'll see a fountain" — predicted a uniform vertical column of cubes. User on Quest reported the opposite.

**Why it happened:**
- Wrote the math without sketching the spatial result on paper
- Tested in Editor where the spatial pattern is harder to see (Editor camera framing)
- Trusted that "random orbit angle + drift up" would naturally produce a column
- Missed that BOTH sources of randomness were keyed on the same index variable, creating perfect correlation

**Fix when caught:**
Pre-compute per-cube random vertical phase offset (deterministic seed) on phase entry — decoupled from orbit angle. Stored in `_vortexVerticalPhase[i]`. UpdateVortex now reads the random offset instead of computing from `i`. Cubes scatter throughout the column at all heights = continuous fountain.

**Rule for next time:**

When two sources of "randomness" both derive from the same index variable, **the result isn't random — it's a deterministic spiral or curve.** Verify pseudo-random distributions are actually independent before assuming they look random in space.

Sketch the spatial result on paper for any procedural placement code BEFORE writing it. Or: compute 5-10 sample points by hand and plot them mentally. Spirals/curves jump out instantly when you do this; they're invisible when you just write the math.

## 3. Trusted my "you're looking the wrong way" theory over the user's eye on the post-vortex collapse

**What happened:**
After P2a vortex ended (state machine moved to P2b_Silhouette), user reported "all the cubes vanish and a small cube sits at the floor." I said:
> *"You're probably looking AWAY from the spawn center. The cubes are still at the original spawn location 1.5m forward. The single cube you see is one outlier."*

User pushed back hard:
> *"i have eyes you think i am lying lol."*

User was right. I was wrong.

**Real bug:** Cubes have a lerp system — `Cube.SetTarget()` records `startPos`, `targetPos`, `lerpStartTime`, `lerpDuration`. `CubeCloudManager.Update()` calls `Tick()` on every active cube every frame, which interpolates `transform.position = Lerp(startPos, targetPos, eased)`.

At spawn, `Spawn(origin)` calls `cube.SetTarget(origin, 0.0001f, Time.time)` — sets targetPos to origin with tiny duration.

My UpdateVortex was DIRECTLY writing `cube.transform.position = newPos` every frame, completely bypassing the lerp system. But I wasn't updating the lerp state — `targetPos` was still the spawn origin from when the cube was first spawned.

When P2a ended and UpdateVortex stopped firing, `Cube.Tick()` kept running every frame in `CubeCloudManager.Update()`. With stale lerp state pointing at the spawn origin and lerpDuration=0.0001, Tick yanked every cube to `targetPos` (the single spawn origin point) within one frame.

Result: ALL 2000 cubes converged to one tiny point on the floor. Looked exactly like "one cube on the floor."

**Fix:**
Sync the lerp state inside UpdateVortex's per-cube loop:
```csharp
cube.transform.position = newPos;
cube.startPos = newPos;
cube.targetPos = newPos;
cube.lerpStartTime = now;
```
Now Tick computes `Lerp(newPos, newPos, ...) = newPos` — no-op, cube stays at last-written position.

**Why I doubted the user:**
- Logs showed `[SpawnRitual] → P2a_Vortex` and `→ P2b_Silhouette` firing correctly
- Logs showed silhouette exception (Read/Write disabled) — I assumed cubes would freeze at last vortex positions because UpdateVortex stopped
- I had a plausible "user looked the wrong way" theory ready
- I overweighted the logs, underweighted the user's clear visual report

**Rule for next time:**

**When the user describes specific visible behavior on the device and the logs disagree, the eyes win.** Logs lie via:
- Stale buffers
- USB disconnects during builds
- Missing log lines for code paths I didn't instrument
- Filtered grep that misses entries

Eyes don't lie about what's on screen.

When I have a plausible "user is mistaken" theory, **test that theory empirically before defending it.** Asking "can you take a screenshot of the spawn area?" would have shown the cubes ARE collapsed to one point. Instead, I lectured about camera angles.

This is the third instance of this pattern in 3 days (per yesterday's mistakes file: "trusted inference over user empirical evidence"). The discipline is **verify before defending**.

## 5. Suggested aggressive bias multipliers without flagging the lighting interaction

**What happened:**
After the round-2 fix attempt with multipliers (1.4 rough, 0.5 metal) showed minimal change, I suggested bumping to (2.5 rough, 0.1 metal):
> *"This is a 30-second test."*

User tried it. Face went DARK / unlit / shadowed.

I should have flagged that very high roughness with NO environment lighting (no HDRI in Reflection Probe) breaks the PBR specular response → diffuse-only fallback → dark surface in shadow regions.

Instead I treated multiplier tuning as an isolated knob with monotonic effect.

**Why it happened:**
- Didn't think through PBR shader behavior at extreme parameter values
- Treated roughness multiplier as a simple linear knob
- Forgot that high roughness is environment-dependent for fill light

**Fix when caught:**
Set `roughnessMultiplier` default back to 1.5 (safe range), added `forceMetallicToZero` boolean for clean zero-out instead of multiplier path.

**Rule for next time:**

For ANY PBR parameter manipulation, ask "what does this look like with NO environment lighting?" before recommending an aggressive value. PBR is environment-dependent by design — high roughness without environment specular is "unlit" not "matte."

Pair any rough/metallic recommendation with the lighting prerequisite: "do this AFTER the Reflection Probe has a Custom HDRI cubemap assigned, otherwise high roughness will look dead."

## 6. Missed the HDRI requirement in initial PBR analysis

**What happened:**
First diagnosis of the founder's plastic complaint identified `forceURPLitFallback` as the smoking gun and proposed:
1. Disable the fallback
2. Add `BiasMaterialsForMatte`
3. Drop dummy asset for shader variant baking

All three are valid. But the FOURTH critical step — **assigning a Custom HDRI cubemap to the Reflection Probe** — was buried as a side note instead of called out as primary.

Cross-AI Round 3 corrected this. Both ChatGPT and Gemini independently called the HDRI step the **dominant** fix (~55% of the plastic resolution per Gemini's probability split). The research agent confirmed: in MR with no skybox, a Realtime Reflection Probe captures the empty Unity scene → black void → PBR specular has nothing to reflect → uniform glossy plastic.

**Why I missed it:**
- I focused on shader-side fidelity (which texture maps survive vs which get stripped)
- I treated the Reflection Probe as a Unity-side default that "just works" once added to scene
- Didn't think about what a Reflection Probe in MR with no skybox actually renders into its cubemap
- Underweighted the lighting environment as a PBR input

**Rule for next time:**

For ANY PBR rendering issue on Quest 3 or any other mobile XR target with no skybox / passthrough background:
1. **Verify the Reflection Probe has a Custom Cubemap with real HDR data** — not Realtime sampling empty space
2. **Treat lighting environment as a first-class PBR input**, not a scene-config afterthought
3. **Test with at least 1 HDRI before claiming any shader-side fix is working**

This is a known limitation of MR + URP that the community has documented. I should have searched it earlier.
