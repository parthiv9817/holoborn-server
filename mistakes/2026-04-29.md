## 5. Recommended "Empty Level" template, didn't realize cube would render unlit

**What happened:**
For the Hello Cube test scene, I told the user to pick UE5's **"Empty Level"** template. They followed the instruction perfectly — empty level, dropped a cube, dropped a Directional Light. Cube rendered with only ONE face lit (the one facing the directional light); other faces went pitch black. User asked "is this a setup problem?" because they'd done everything I said.

**Why it happened:**
- "Empty Level" is *literally empty* — no Sky Atmosphere, no Sky Sphere, no Sky Light fill. UE5's default ambient is 0; without an ambient source, faces away from the directional light have zero illumination.
- I told the user to "add a Sky Light" as the fix — but Sky Light's default `Source Type = SLS Captured Scene` captures the surrounding scene as its ambient source. In Empty Level, the surrounding scene is *also* black void. Sky Light captured black, added zero fill. Cube stayed dark.
- The right level template for a quick visual test is **Basic** — ships with Sky Atmosphere + Sky Light + Directional Light + Floor + Fog + Volumetric Cloud, all pre-configured to "look like a 3D scene." Goldilocks tier.

**Rule for next time:**
**When recommending a UE5 level template for "render something quickly to confirm toolchain works," default to Basic, not Empty.** Empty is for engineers who know they want to assemble lighting from zero. Basic gives you working PBR illumination on day zero. The template names are misleading — "Empty" sounds neutral / minimal but is actually the most hostile to first-test workflows.

More generally: **verify a template has lighting infrastructure before recommending it for a visual test.** Open it once myself, look at the world outliner, count actors. If actors == zero, the test will look broken.

## 6. Described a glTFRuntime "import factory" workflow that doesn't exist

**What happened:**
When the user asked how to import the GLB into UE5, I confidently described a workflow: "drag the .glb into Content Browser → glTFRuntime registers a custom factory → pick `glTFRuntimeAsset` over the standard Static Mesh import → asset is wrapped as a `UglTFRuntimeAsset` cooked into the APK." The user followed the steps and saw only Interchange's "Default GLTF Assets Pipeline" options — no glTFRuntime factory. Pushed me to web-search.

The web search confirmed: **glTFRuntime is purely a RUNTIME plugin. It never had an editor import factory.** Its API is `glTFLoadAssetFromFilename / FromUrl / FromData` — load at runtime, return a `UglTFRuntimeAsset` that lives in RAM only, never gets cooked. The whole point is loading GLBs the project doesn't know about at compile time (downloaded from server, generated by AI, etc.). Editor-time import is solved by Interchange or the legacy glTF Importer plugin — completely separate path.

**Why it happened:**
- I generalized from how *some* UE5 plugins work (registering custom factories alongside built-ins) to how glTFRuntime works specifically. Bad pattern-matching.
- Yesterday's roadmap research mentioned glTFRuntime would be in `Plugins/` and called from Blueprint, but didn't specify whether the editor workflow includes asset import. I filled the gap with a plausible-sounding made-up workflow.
- Didn't actually open glTFRuntime's README or recent forum posts before describing the workflow. Cost of checking: 60 sec. Cost of being wrong: ~15-20 min of user follow-on work + a credibility hit.

**Rule for next time:**
**For any plugin, verify the actual usage pattern against the plugin's README or current docs before describing the workflow.** Don't assume plugin patterns generalize. Especially don't assume "import factory exists" just because drag-and-import feels like a natural UX. The cost of a 30-second README check pays for itself many times over compared to the cost of being confidently wrong.

When a user pushes back ("are you sure?"), don't double down — go verify. The user pushed back here, and they were right to. Reward that instinct, don't fight it.

## 8. Didn't explain Blueprint exec-pin vs data-pin distinction upfront

**What happened:**
Walking through the Level Blueprint setup, I told the user "drag from Event BeginPlay's exec pin to Get Player Controller." User followed exactly — dragged the exec wire (white triangle) and released it on Get Player Controller's `Player Index` input pin (which is an integer-typed data pin, NOT an exec pin). UE5 errored with "Exec is not compatible with Integer." User pinged confused.

**Why it happened:**
- I assumed the exec-pin vs data-pin distinction was implicit in the visual cues (white triangles vs colored circles), but UE5's UI doesn't make this obvious enough for a first-time Blueprinter to grok without explanation.
- I specified the SOURCE pin (Event BeginPlay's exec, white triangle) but didn't specify the DESTINATION pin requirement (must also be an exec pin, which Get Player Controller doesn't have because it's a pure function).
- The "pure function vs executable function" concept was buried in my explanation rather than called out as a distinct mental model upfront.

**Rule for next time:**
**Before walking a Blueprint-newcomer through their first wire, teach the exec-pin vs data-pin distinction explicitly.** Two-line crash course:
- White triangles = "what to do next" (program flow)
- Colored circles = "data values" (typed: integer, float, object reference, etc.)
- They never connect to each other. Same shape and color = compatible.

Then: pure functions (Get X, Is Y, Find Z) have NO exec pins because they don't change state. Executable functions (Spawn Actor, Add Mapping Context, Print) DO have exec pins. The chain divides into "data flow" (sideways) and "exec flow" (forward), meeting at the executable function.

This is the kind of foundational concept that should be a 30-second preface to ANY Blueprint walkthrough, not a remediation after the user errors.
