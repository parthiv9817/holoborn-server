# 2026-05-29 — Mistakes

A clean, productive day (decimation solved + wired), but five instructive mistakes — most of them mine (Claude's) in execution. Themes: trusting a tool's defaults without checking the renderer's support, picking the wrong artifact before reading the constraint, ambient-environment assumptions, and register-matching in founder comms.

---

## 1. Ran gltfpack with default quantization → scrambled the textures

**What happened:**
First decimation pass was `gltfpack -si 0.18` with default settings. Tri count dropped correctly (452k → 81k), but on Quest the textures rendered as **random scattered patches** — body geometry intact, UVs destroyed. Parthiv pulled the capture; clear UV-corruption signature. Diagnosed via the GLB JSON: `extensionsUsed: [KHR_mesh_quantization, KHR_texture_transform]`, `TEXCOORD_0` quantized to normalized UNSIGNED_SHORT. gltfpack's default quantization + auto-injected `KHR_texture_transform` is not applied correctly by glTFast 6.18 → UVs map wrong. Re-ran with `-noq` → plain FLOAT UVs, no extensions, same 81k tris, textures correct.

**Why it happened:**
- I reached for gltfpack's defaults assuming "fewer tris" was the only variable. Quantization is a *separate* size optimization that the **renderer must support** — and I didn't check what this Unity project's glTFast actually supports before trusting the output.
- Earlier in the day I'd even confirmed the Unity manifest has glTFast 6.18 with NO KtxUnity/meshopt packages — but I didn't connect that "limited extension support" to "quantization might also break."

**Rule for next time:**
**`-noq` is mandatory for output bound for glTFast 6.18.** More generally: before trusting *any* gltfpack flag, check `extensionsUsed` on the output and confirm the consuming renderer supports every extension emitted. Quantization/KTX2/meshopt all add extensions that silently fail to load if the runtime lacks the package. The `-si` triangle reduction is independent of all of them.

## 2. Almost validated decimation on the wrong (post-rig) artifact

**What happened:**
I baselined `test_rigged.glb` (471k, already rigged) and was set to test decimation on it. Parthiv corrected: decimation must run **pre-rigging**, because simplifying a rigged mesh resamples the per-vertex skin weights and breaks deformation. Switched to the pre-rig retex GLB (skins=0).

**Why it happened:**
- I picked the most-recently-touched GLB as the test subject without first establishing *where in the pipeline* decimation belongs.
- The `REFINEMENT-PHASE.md` doc literally said "before Meshy rigging so weights stay clean" — I had the constraint and didn't apply it before choosing a target.

**Rule for next time:**
**Establish the pipeline position of a transform BEFORE choosing a test artifact.** Decimation goes post-retex / pre-rig → validate on the retex GLB (skins=0), never the rigged one. Read the constraint, then pick the input.

## 3. Restarted uvicorn from the wrong working directory → ModuleNotFoundError

**What happened:**
When restarting a fresh uvicorn (to clear any cached state), it crashed at boot: `ModuleNotFoundError: No module named 'app'`. The tunnel 502'd, local serve dead. The *first* uvicorn start that day had worked only because the shell's cwd happened to be the repo root; the restart launched from elsewhere. Fixed by pinning `PYTHONPATH=<repo root>` (and cd).

**Why it happened:**
- Relied on ambient working directory for module resolution. Background/relaunched processes don't guarantee cwd, and `app.main:app` is only importable from the repo root.

**Rule for next time:**
**Always launch uvicorn with an explicit `PYTHONPATH=<repo root>` (or `--app-dir <repo root>`).** Never depend on ambient cwd for the `app` package import — especially for backgrounded launches.

## 4. Drafted the founder update in production register for a POC

**What happened:**
First draft of the Vipin update said "before I turn it on for **production**… the only thing between this and **shipping**." Parthiv flagged that HoloBorn is a **POC**, not production — the framing overclaimed maturity. Rewrote in POC register ("the demo walk-around is smooth," "off by default — nothing changed in the current flow," "ready to switch on").

**Why it happened:**
- I defaulted to generic SWE shipping-language without matching it to where the project actually is in its lifecycle.

**Rule for next time:**
**Match register to project maturity in stakeholder comms.** POC ≠ production. Say "demo," "ready to switch on," "validated" — not "shipping to production" — unless it genuinely is in production.

## 5. brew install gltfpack — red herring

**What happened:**
Kicked off `brew install gltfpack` assuming a Homebrew formula existed. It doesn't (brew only surfaced an unrelated cask, `gltfquicklook`). Wasted a background install before switching to the correct route: `npm i -g gltfpack`.

**Why it happened:**
- Assumed Homebrew coverage for a tool that ships primarily via npm.

**Rule for next time:**
**gltfpack installs via npm (`npm i -g gltfpack`), not Homebrew core.** Verify the install route (or just check `npm`/official releases) before firing a package-manager install.

## Pattern across the five
Three of them (#1, #2, #3) share a root: **trusting a default/ambient state instead of verifying the actual constraint** — gltfpack's defaults, the cwd, the artifact's pipeline position. The constraint information was *available* in each case (the Unity manifest, the refinement doc, the import path); the mistake was acting before reading it. The fix is the same shape as the 05-25 lesson: name what the thing depends on before you touch it.
