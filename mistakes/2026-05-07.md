## 4. Wrong API hypothesis with `InstantiationSettings.AnimationMethod` — guessed instead of grepped

**What happened:**
When fixing the frozen-avatar issue (animations not playing), I confidently recommended:
```csharp
var instantiationSettings = new InstantiationSettings { AnimationMethod = AnimationMethod.Legacy };
```

Compile error: `'InstantiationSettings' does not contain a definition for 'AnimationMethod'`. Had to fix to use `ImportSettings` instead. One build cycle wasted on the wrong API surface.

The actual fix was:
```csharp
var importSettings = new ImportSettings { AnimationMethod = AnimationMethod.Legacy };
await gltf.LoadGltfBinary(bytes, null, importSettings);
```

After the compile error, I grepped `Library/PackageCache/com.unity.cloud.gltfast@.../` and found `AnimationMethod` was a property on `ImportSettings`, not `InstantiationSettings`. 30 seconds of grep would have caught this BEFORE I wrote the wrong code.

**Why it happened:**
- glTFast's API has both `ImportSettings` and `InstantiationSettings`; both are `[Serializable]` classes; both control behavior. Confused them.
- Wrote code based on plausible-feeling API shape rather than verified shape
- Didn't run AST parse / compile check before declaring "this should work"

**Rule for next time:**
**For any API I don't have memorized, grep the actual source BEFORE writing code that depends on the guess.** Cost of grep: 30 seconds. Cost of wrong-API-call build cycle: 1-3 minutes minimum + the cognitive overhead of reading the compile error and re-doing the fix.

Concretely for glTFast / Unity packages: source lives at `Library/PackageCache/<package-name>@<hash>/Runtime/Scripts/`. Always grep there before writing API calls, especially for properties/settings I'm guessing at.

This is similar to 2026-05-04 mistake #1 (declared root cause before verifying) — both are "asserted before checking when verification was cheap." Today's version had a fast feedback loop (compile error within 30s) so the cost was small, but the principle is the same.

## 5. Started troubleshooting backend serving when ngrok bandwidth limit was the actual blocker

**What happened:**
When `curl -sI` to the ngrok URL returned `HTTP/2 403`, my first instinct was to start checking backend serving — was uvicorn running, was the static mount correct, was the file path right. Started running diagnostic commands.

Then I read the actual response body. The HTML had `ERR_NGROK_725` and the literal text *"This ngrok account has reached its network bandwidth limit for the month."* That was the entire diagnosis sitting in the first response.

Wasted ~30 seconds running ps aux + curling the URL with different headers + checking config before reading what ngrok was telling me directly.

**Why it happened:**
- Pattern-matched "HTTP error → check the server" without reading the error body
- ngrok's 403 is unusual — most 403s come from the application, not the tunnel
- Default mental model: "if backend is up, file should serve" — didn't account for the tunnel itself blocking

**Rule for next time:**
**Read error response bodies before dispatching to debug commands.** Especially for ambiguous status codes like 403. The error message often IS the diagnosis.

Concretely: when curl returns 4xx/5xx, ALWAYS dump the response body first via `curl -s URL` (no `-I`, get the body). Look at the HTML/text. Most error pages tell you WHICH layer of the stack failed. Then dispatch to that layer's diagnostics.

For ngrok specifically: ngrok's error pages are well-formatted and ALWAYS include the ERR_NGROK_XXX code which directly maps to a docs page (https://ngrok.com/docs/errors/err_ngrok_725 etc). Grep these codes; their docs are the definitive answer.

## 7. `Mesh.RecalculateTangents` post-spawn was probably actively harmful

**What happened:**
Added defensive code in TestGlbLoader.cs that walked all SkinnedMeshRenderers post-spawn and called `Mesh.RecalculateNormals + RecalculateTangents + RecalculateBounds`. Hypothesis was: Meshy's auto-rig might have stripped tangents during export, so re-computing them would help URP normal-map sampling.

Research agent flagged this in the dispatched research report (around 17:00):
> *"`RecalculateTangents` was here — removed 2026-05-07 per research finding: forcing tangent recompute against Meshy's exported tangents creates a mismatch with the normal map's tangent-space convention (MikkTSpace assumed). Better to leave glTFast's imported tangents alone."*

`RecalculateNormals()` computes normals from triangle winding. If winding is broken/inconsistent (which Meshy's export apparently has — that's why we needed `_Cull = 0`), recomputed normals are ALSO broken/inconsistent — locking in the bad state.

**Why it happened:**
- ChatGPT's first analysis suggested tangent issues, "tangent recompute" felt like the obvious fix
- Didn't consider the interaction: recomputed tangents must match the normal-map's authoring convention (MikkTSpace), but Mesh.RecalculateTangents uses a different algorithm
- "Defensive" code that I labeled in comments as "cheap, helps with normal-map sampling artifacts" without verifying it actually helped

**Rule for next time:**
**Don't add "defensive" code that mutates imported assets without verifying the mutation actually helps.** Specifically for mesh data:
- `Mesh.RecalculateNormals` only helps if the mesh has WRONG normals AND consistent winding (it computes from winding)
- `Mesh.RecalculateTangents` only helps if the normal-map authoring convention matches Unity's tangent algorithm
- Most exporters that include normals/tangents already have them right; recomputing risks breaking them

Concretely: if the symptom is "wet/oily during animation" and the structural data (verts, UVs, weights) imports correctly, the issue is more likely on the rendering side (shader, lighting, culling) than on the mesh-data side. Skip the mesh recompute calls unless there's specific evidence the imported data is wrong.
