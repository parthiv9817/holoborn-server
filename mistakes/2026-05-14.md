## 1. Cancelled the first real e2e run mid-cold-start

**What happened:**
Parthiv's first real-backend run got stuck IN_QUEUE on RunPod for ~11 minutes. I assumed it was zombied, called `cancel` on the job. Cancel response came back showing `delayTime=764s, executionTime=13.7s, workerId=ytykn9nyt5d1h4` — meaning a worker had JUST picked it up at the moment I cancelled. The job was 13 seconds into actually running.

If I'd waited another 3-5 minutes, TRELLIS would have completed normally. Instead I killed Parthiv's first real attempt at the e2e and we had to restart from scratch (validate → burst → portraitize → TRELLIS again).

**Why it happened:**
- I conflated "long queue wait" with "stuck/dead job" without checking executionTime in the status payload.
- Queue waits on RunPod's free-tier endpoint can be 5-12 minutes from cold (workersMin=1 was JUST set 20 min before, the worker hadn't fully cold-started yet).
- I felt urgency to "do something" rather than wait. False productivity.

**Rule for next time:**
**Before cancelling a queued RunPod job, check status payload's `delayTime` AND `executionTime`.** If executionTime > 0, the job has been picked up — let it run. If delayTime is climbing past the documented cold-start window (~9 min), THEN consider cancel. Default to waiting longer than feels comfortable.

Compound learning: "I should DO something to fix this" is often wrong. The system's already doing the right thing; my impatience is the bug.

## 2. Shipped HUD-follow without checking aesthetic with user

**What happened:**
Pipeline progress indicator hit a placement bug — column locked at first-spawn camera position, user turns head, column off-screen. My fast fix: track the camera every LateUpdate (HUD-style). Shipped without asking.

Founder verdict on screenshot: "this is so annoying and felt cheap". Council audit independently agreed: HUD-follow shatters MR immersion, reads as Unity XR debug overlay. We then had to spend ANOTHER 2-3 iterations pivoting away from HUD-follow → 360° ring → final Diagnostic Crescent.

I shipped a fix that solved the technical problem (visibility) at the cost of the aesthetic goal (ritual instrumentation, not interface chrome). The two were in tension and I picked the wrong axis without consulting.

**Why it happened:**
- Tunnel-vision on the visibility bug. "Column goes off-screen" felt like a hard error so I picked the obvious fix.
- Didn't pause to ask "but does the user WANT it always visible, or is finding-it-by-looking the actual product behavior?"
- The product framing (scientist examining biosymbiote specimen — ambient world-element, not bolted UI) was already locked from morning's brainstorm. I had access to it. I ignored it under deadline pressure.

**Rule for next time:**
**When facing a tension between "always visible" and "embedded/world-locked", surface the choice to the user before shipping.** Especially on UX-load-bearing decisions in a product where aesthetic IS the product. Premium MR explicitly trades constant visibility for spatial dignity. HUD-follow defaults to the wrong side of that tradeoff.

Compound learning: tunnel-vision under deadline pressure → shipping the technically-correct-but-aesthetically-wrong fix. The fast fix that has to be undone burns more time than the slower right one.

## 3. V3 portraitizer prompt's hardcoded "lanyard with badge" caused hallucination

**What happened:**
First real e2e produced an avatar wearing a blue lanyard + ID badge — but the test subject in the input photo was NOT wearing one. GPT Image 1.5 invented it.

Root cause: V3 prompt literally said
> "same lanyard around the neck with the same badge attached"

as part of the clothing preservation clause. Left over from earlier dev iterations where the operator always wore a Digispoc badge during testing. GPT Image 1.5 reads the prompt as ground truth: "I'm told to preserve a lanyard that should be there → if I don't see one in the input, the input must be incomplete → I'll render one to match."

Self-contradiction also slipped past prior review: the same prompt later said "Do not add accessories, text, logos, or watermarks that are not present in the input image." The earlier "same lanyard" clause overrode it.

**Why it happened:**
- Specific-item enumeration in prompts is a known footgun for diffusion models, especially edit-mode where the model already has strong priors about what "should be there".
- The V3 prompt was tuned during a period when the test pool happened to all wear badges. The lanyard line worked for that pool. Nobody flagged it when the pool broadened.
- No automated visual regression test compares raw input vs portrait output for hallucinated additions.

**Rule for next time:**
**Don't enumerate specific clothing items in image-edit prompts.** Use generic "preserve every item visible in the input, NEVER add items not visible" rules. Always include explicit negation: "If X is not visible, do NOT render X." Diffusion models respond to the presence of a term — listing it makes the model want to render it.

Compound learning: prompts should describe the TRANSFORMATION ("preserve everything visible") not the EXPECTED OUTPUT ("they should have a lanyard"). The latter biases toward hallucination on out-of-distribution inputs.

## 4. Inner exception clause didn't catch the actual exception type

**What happened:**
First Meshy Rigging attempt timed out at 60s on the POST. I'd written the inner try/except as:
```python
except (MeshyJobError, MeshyDownloadError, TimeoutError) as e:
    # graceful fallback to retex as final
```

`httpx`/`httpcore` raise `httpcore.ReadTimeout` for network timeouts, which extends `httpx.HTTPError` — NOT Python's built-in `TimeoutError`. The exception leaked past the inner except, was caught by the OUTER `except Exception`, which marks task=failed and returns. Quest UI never got "complete" status; user saw the ritual hang at "Adding Bones".

The graceful fallback I'd written explicitly to handle this case never ran. The bug was in the exception narrow-catch.

**Why it happened:**
- I conflated Python `TimeoutError` (built-in, used for asyncio.wait_for etc.) with `httpx.TimeoutException` (httpx network timeouts). They're in different inheritance trees.
- Didn't test the failure path before shipping (only tested the success path mentally).
- Documentation on which exception type httpx raises for which scenario is buried.

**Rule for next time:**
**For HTTP client error handling, catch `httpx.HTTPError` as the parent class** — covers timeouts, network errors, status errors, decode errors. Don't try to enumerate specific subtypes. If you must distinguish, layer specific catches BEFORE the parent.

Compound learning: when you write a graceful fallback, TEST IT by inducing a failure. Mock the upstream service to raise; verify the fallback path fires. "I wrote the except block, surely it works" is the bug.

## Pattern across today's mistakes

**The throughline:** I optimized for FAST closure over RIGHT closure in three of the five mistakes (#1 cancel, #2 HUD-follow, #4 exception). Each one ended up costing more time than the patient path would have.

Specific manifestations:
- Cancel a "stuck" job to "do something" → killed Parthiv's first attempt and lost ~20 min
- Ship a HUD-follow to "fix" visibility → forced 3 more pivot iterations to undo
- Write a narrow exception catch to "be precise" → silently failed the fallback path

The pattern: under deadline pressure, fast-closure looks productive. But the work it forces downstream is invisible at decision-time. Patient closure feels slow but compounds.

**Rule that addresses all of these:**

> Before shipping a "fix" under deadline pressure, ask: am I solving the right problem? Have I checked all the failure modes my fix creates? Am I optimizing for closure-speed at the expense of work-quality? If yes, slow down — the slower path is faster on net.
